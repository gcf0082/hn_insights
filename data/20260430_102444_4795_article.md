# 洞察报告：Caveman vs "be brief." 基准测试

**来源链接**：https://www.maxtaylor.me/articles/benchmarked-caveman-against-two-words
**原始标题**：I benchmarked caveman against two words
**作者**：Max Taylor
**发布日期**：2026-04-29
**阅读时间**：5分钟

## 核心发现

这是一项针对 Claude Code 压缩插件 **Caveman** 的系统性基准测试，将其在三种强度等级（lite、full、ultra）下的表现与简单的两个字指令 **"be brief."** 进行对比。

### 关键结论

1. **质量未受影响**：所有测试组的质量得分均在 97.0%-98.5% 之间，相差不超过 1.5 个百分点。每个测试组都覆盖了 100% 的关键要点，120 次响应中零次触发"必须避免"陷阱。

2. **Token 消耗相近**：
   | 测试组 | 平均 Token 数 |
   |--------|-------------|
   | baseline | 636 |
   | **brief** | **419** |
   | lite | 401 |
   | full | 404 |
   | ultra | 449 |

   "Be brief." 将 token 减少了 34%，与 Caveman lite 和 full 水平相当。

3. **Ultra 的矛盾表现**：Ultra（最严格模式）在总体上产生了最长的回答，但在 bug 诊断、概念解释和错误解读类别上却最短。原因是 Caveman 内置了"Auto-Clarity"规则——在这些高风险类别上自动停止压缩。

## Caveman 的真正价值

测试表明，如果用户只想要更简短的输出，"be brief." 指令已经完全够用。Caveman 的差异化价值在于：

- **一致的输出格式**：每次响应都遵循统一的 `[问题] → [解决] → [原因]` 模式
- **强度调节拨盘**：可通过 slash 命令随时切换 lite/full/ultra 模式
- **长对话中的规则保持**：通过 SessionStart 和 UserPromptSubmit 钩子持续重新注入规则集

## 值得关注的发现

- **Lite 遗漏术语**：在一次 SQS vs BullMQ vs Kafka 对比问题中，lite 的紧凑格式导致遗漏了"at-least-once"这一必需术语，得分 0.70（全场最低）
- **Ultra 触发工具调用行为**：Ultra 在处理 Dockerfile 设置问题时尝试调用 Write 工具，产生约 1300 token 的额外输出

## 实际建议

- **只需简短输出** → 使用 `"be brief."` 写在 prompt 或 CLAUDE.md 中
- **需要统一输出风格或下游工具集成** → 使用 Caveman

**核心启示**：大多数提示工程建议从未与默认设置进行对比测量。动手测量它。
