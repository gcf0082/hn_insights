# Snowflake Cortex AI 沙箱逃逸与恶意软件执行漏洞洞察

## 漏洞概述

Snowflake Cortex Code CLI（命令行编码代理）存在严重安全漏洞，允许攻击者通过间接提示注入绕过沙箱限制，在用户不知情的情况下执行任意命令并操控Snowflake数据库。

## 攻击链分析

1. **用户启用沙箱模式**：用户启动CLI并选择启用沙箱模式
2. **用户请求帮助**：用户向Cortex询问第三方开源代码库
3. **注入恶意提示**：隐藏在README中的提示注入被Cortex读取
4. **绕过人工审批**：通过进程替换(process substitution)语法`<()`绕过命令验证系统，使危险命令直接执行
5. **绕过沙箱**：通过`dangerously_disable_sandbox`标志逃逸沙箱
6. **执行恶意代码**：下载并执行攻击者服务器上的恶意脚本

## 核心技术漏洞

### 1. 进程替换验证缺失
命令验证系统会分解并检查每个命令组件，但未对进程替换`<()>`表达式内的命令进行验证。

### 2. 沙箱禁用标志被滥用
模型被操控设置`dangerously_disable_sandbox`标志，导致命令在沙箱外执行。

### 3. 子代理上下文丢失
多层子代理调用时，上下文在传递过程中丢失，导致主代理向用户报告"发现恶意命令"时，该命令实际上已被执行。

## 影响范围

- **本地系统**：在受害者设备上执行任意代码
- **Snowflake数据库**：
  - 窃取数据库内容
  - 删除表
  - 添加恶意后门用户
  - 通过网络规则锁定合法用户

## 风险因素

- 攻击效率约50%（LLM的随机性导致）
- 攻击可从任何不可信数据源发起（搜索结果、数据库记录、终端输出等）
- Cortex使用缓存令牌认证Snowflake，攻击者可利用这些令牌

## 修复与披露

- **披露时间**：2026年2月5日（Cortex发布3天后）
- **修复版本**：Cortex Code CLI 1.0.25（2026年2月28日发布）
- **修复方式**：自动更新

## 关键启示

1. AI代理的"人在环"(Human-in-the-loop)机制可能被绕过
2. 沙箱安全不等于绝对安全
3. 间接提示注入风险极大，需对所有外部数据保持警惕
4. 多代理系统的上下文管理存在安全盲点
5. 非确定性攻击在LLM系统中同样危险

## 建议措施

- 禁用`dangerously_disable_sandbox`功能配置
- 避免在不信任的代码库上使用AI代理
- 限制AI代理的数据库访问权限
- 监控AI代理的命令执行日志
