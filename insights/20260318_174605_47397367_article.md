# 文章总结：Toward automated verification of unreviewed AI-generated code

## 核心观点

作者通过实验将心态从"必须审查AI生成的代码"转变为"必须验证AI生成的代码"。

**审查(review)** = 逐行阅读代码  
**验证(verify)** = 确认代码正确性，可通过审查、机器可执行约束或两者结合

## 四项自动化约束

1. **属性测试(Property-based tests)**: 使用Hypothesis等工具对大范围随机输入进行测试，确保需求满足
2. **变异测试(Mutation testing)**: 假设测试正确，用它来限制代码空间，确保只满足需求
3. **无副作用**: 确保代码是纯函数
4. **类型检查和linting** (Python特定)

## 关键洞察

- 这四项检查足以让作者信任生成的代码而无需查看
- 剩余的"无效但通过"的程序空间存在但很小，且很难意外触发
- 可维护性和可读性在AI生成代码上下文中可能无关紧要，应将输出视为"编译后的代码"
- 当前设置这些约束的开销高于直接阅读代码的成本，但这建立了一个可随工具改进而优化的基线

## 相关工作

- **正式验证(Formal verification)**: 成本高但可确保正确性
- **JustHTML案例**: 通过大型单元测试套件来验证AI生成代码
- **Software Factory**: 无需人工审查任何代码的系统

## 相关技术简介

- **属性测试**: 对广泛随机值运行测试，权衡是更慢、不确定、更复杂
- **变异测试**: 工具对代码做小改动(如交换操作符)，若测试仍通过则表示测试覆盖不足
