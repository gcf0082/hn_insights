# HN Insights: AI Coding Is Gambling

**Source**: https://news.ycombinator.com/item?id=47428541  
**Date**: 2026-03-18  
**Score**: 72 points

## 核心观点

### 1. AI编码本质是概率性赌博
- LLM输出是概率性的（probabilistic），本质上是"投注"tokens期望获得期望的输出
- 成功与否不可完全控制，有时AI会假装能处理问题（hallucination）

### 2. 成瘾风险比技术风险更值得警惕
- 快速反馈循环让人难以停止使用（"token anxiety"）
- 有人反映使用AI后工作到凌晨2点，难以设定边界
- 有人将其比作老虎机同时提供准社会关系（parasocial relationship）

### 3. 使用方式决定风险程度
- **高风险**: 快速一次性的"spray and pray"式vibe coding
- **中等风险**: 规范驱动 + TDD + AI辅助，类似扑克游戏
- **低风险**: 配合严格的项目管理和测试实践

### 4. 团队实践案例
- 有团队采用spec-driven + TDD开发模式，开发者被禁止直接写代码
- 配合Cursor + Speckit + Notion + Jira的工作流
- 结果表明有效，但工具链搭建需要前期投入

### 5. 关键限制
- 随着代码库增大，需要更多tokens来保持低错误率
- 模型本身不会从你的使用中学习（参数不会改变）
- 保持上下文相关性是持续挑战

### 6. 与传统编程对比
- 人类编程本身也涉及概率性（"80% chance that worked"）
- 有人将AI比作"超级能干的实习生"——有非确定性但有潜力
- 关键区别：实习生可以学习成长，AI不会

## 社区共识

多数人认为：AI编码本身不是问题，**问题在于使用方式**。配合规范定义、测试驱动和适当的检查机制，AI可以是高效工具；盲目依赖而不验证则是真正的"赌博"。
