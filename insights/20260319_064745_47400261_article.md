# MCP Server正在消耗你的上下文窗口。还有更简单的方法

**来源**: https://www.apideck.com/blog/mcp-server-eating-context-window-cli-alternative  
**日期**: 2025年

## 核心问题

MCP (Model Context Protocol) 工具定义会消耗大量token:
- 连接3个服务(GitHub、Slack、Sentry)，约40个工具，在AI处理任何用户消息前，上下文窗口已消耗**55,000+ tokens**
- 单个MCP工具消耗 **550-1,400 tokens**(名称、描述、JSON Schema、字段描述、枚举、系统指令)
- 某团队报告：3个MCP服务器消耗143,000/200,000 tokens (72%)

## 三种解决方案对比

### 1. MCP + 压缩技巧
- 压缩Schema、按需加载工具定义、构建中间件切片OpenAPI规范
- 适用于小范围、高频交互
- 需要额外的基础设施：工具注册表、搜索逻辑、缓存、路由

### 2. 代码执行
- 代理像开发者一样，有持久工作空间
- 代理需要新集成时：阅读API文档、编写SDK代码、运行、保存脚本
- 适用于长时间工作的workspace代理，需要复杂工作流(循环、条件、轮询、批处理)
- Cloudflare的Code Mode：代理编写短编排脚本调用结构化MCP工具

### 3. CLI作为代理接口(推荐)
- **渐进式披露**：代理通过`--help`发现能力，而非预先加载所有schema
- token消耗对比：
  - 完整OpenAPI spec: 30,000-100,000+ tokens
  - MCP工具(~3,600/API): 10,000-50,000+ tokens
  - CLI代理prompt: **~80 tokens**
  - CLI `--help`调用: 50-200 tokens(仅在需要时)

## CLI的优势

### 渐进式披露节省token
```bash
# 80 tokens的代理prompt
Use `apideck` to interact with the Apideck Unified API.
Available APIs: `apideck --list`
List resources: `apideck <api> --list`
Operation help: `apideck <api> <resource> <verb> --help`
```

代理分三层发现能力：
1. `apideck --list` → 列出可用API (~20 tokens)
2. `apideck accounting --list` → 列出资源 (~200 tokens)
3. `apideck accounting invoices create --help` → 查看操作帮助 (~150 tokens)

### 可靠性：本地优于远程
- Scalekit基准测试：MCP调用GitHub Copilot服务器**28%失败率**
- CLI无此问题：二进制文件本地运行，无远程服务器超时

### 结构化安全优于提示词安全
- 权限分类内置于二进制：
  - GET → 自动批准
  - POST/PUT/PATCH → 需要确认(`--yes`)
  - DELETE → 默认阻止(`--force`)
- 提示词注入无法覆盖此安全机制

### 通用兼容性，零协议开销
- 所有代理框架都支持"运行shell命令"
- MCP需要专用客户端支持、连接管理、服务器生命周期管理

## Scalekit基准测试数据(75次对比)

| 指标 | CLI | MCP |
|------|-----|-----|
| token消耗 | 1,365 | 44,026 |
| 成本(每月10000次操作) | $3.20 | $55.20 |
| 失败率 | ~0% | 28% |

## 何时不适用CLI

- **MCP适合**：紧密范围、高频工具(5-10个工具调用数百次)
- **代码执行适合**：复杂、有状态的工作流(轮询、聚合、跨分页、带回滚的多步事务)
- **MCP适合**：代理代表他人用户时(需要OAuth和动态客户端注册)
- **限制**：流式和双向通信、跨平台分发

## 对API提供商的建议

1. OpenAPI spec太大：50+端点会消耗大部分代理预算
2. 渐进式披露不仅是UX模式：也是token优化策略
3. 结构化安全不可协商：内置权限模型，而非依赖提示词
4. 机器友好输出格式：非交互环境默认JSON、稳定退出码
