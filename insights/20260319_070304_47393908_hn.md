## 核心定义

Simon Willison 将 **Agentic Engineering** 定义为：

> "借助 coding agents（能够编写和执行代码的智能体）辅助开发软件的实践"

**Coding agents** 的核心特征：
- 能同时编写和执行代码
- 示例：Claude Code、OpenAI Codex、Gemini CLI
- 通过"在循环中调用工具"来实现目标

## 社区讨论要点

### 1. 命名争议：新技术还是新学科？

**质疑方观点**：
- "这不算新学科，只是软件工程中的一种新技术"
- 传统软件工程实践（测试、模块化、反馈）仍然适用
- 命名可能暗示"engineering of agents"而非"engineering using agents"

**支持方观点**：
- 需要一个术语来区分"vibe coding"和认真负责的代码生成
- 这是行业专业化的必然演进

### 2. Vibe Coding vs Agentic Engineering 的边界

Simon Willison 的定义：
- **Vibe coding**：prompt 但从不查看生成的代码
- **Agentic engineering**：开始关注代码并承担责任

关键区分：**对代码负责**，停止将错误归咎于 LLM

社区提出的不同参与度比例（maxbond 提出）：
- 80%+：不理解代码库，靠手动测试确保正确性
- 50%：理解代码结构，检查变化，代码质量可疑
- 20%-：设计代码结构，自己编写大部分代码

### 3. 责任与问责

- LLMs 无法被问责，必须有人类承担责任
- 问责的关键：愿意为代码承担全部责任
- 机械/土木工程师虽然不亲自动手建造，但必须深入理解设计的物理/数学特性

### 4. 批评与担忧

**技术层面**：
- 非确定性带来可观测性挑战
- 缺乏标准化的 agent 行为视图
- 需要关注"幂等性"——agent 重启时每个操作都会重放

**安全层面**：
- 预言：会发生类似 Challenger 灾难的 coding agent 安全事件
- Amazon 因 AI 代码问题导致数百万订单损失
- 可能会看到重大 AI 相关安全事件

**行业层面**：
- 存在"严重 delusional state"
- 工具被过度宣传为"灵丹妙药"
- 需要区分"在有限范围内有用"和"可以替代一切"

### 5. 实际应用洞察

**有效实践**：
- 提供清晰的目标和详细的规格说明
- 让 coding agents 迭代直到目标达成
- 创建小型确定性工具，组合使用效果更好
- 传统工程实践（测试、lint、CI/CD）能帮助 agents 产出更好结果

**反模式**：
- 未经审查的代码强加给协作者
- 过度依赖 agents 而不验证结果
- 将 agents 吹捧为什么都能做的"黑箱"

## 总结

社区对"agentic engineering"这一术语存在分歧：

| 观点 | 代表声音 |
|------|----------|
| 这是营销术语，用来区别于 vibe coding | 质疑者 |
| 这是软件工程工具的自然演进 | 支持者 |
| 需要更好的可观测性和责任机制 | 安全关注者 |
| 过度炒作，实际能力有限 | 批评者 |

核心共识：**无论叫什么名字，工程师必须为最终代码负责**——传统工程实践的价值在 AI 辅助时代不减反增。
