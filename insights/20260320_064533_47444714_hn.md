# No AI in Node.js Core

## 背景

2026年1月，一个19,000行的PR被提交到Node.js核心，提交者（Node.js长期贡献者mcollina）在PR描述中声明使用了大量Claude Code tokens来创建此PR。这一事件引发了对AI辅助开发是否应被允许的讨论。

## 主要观点

### 反对AI代码进入Node.js核心

1. **代码质量担忧**：AI生成的代码看起来很好，但不一定正确。需要更多审查， 给维护者增加负担。

2. **规模问题**：AI可以大规模创建PR，可能淹没维护者，排挤非AI的PR。

3. **DCO合规性问题**：根据开发者来源证书，贡献必须由人创建，AI辅助是否满足这一要求存在争议。

4. **可复现性要求**：提交的生成代码应该可以让审查者无需通过付费AI工具即可复现。

5. **Node.js是关键基础设施**：Node.js运行在数百万服务器上，核心代码应该经过精心手写和维护。

### 支持或中立观点

1. **工具中立**：只要提交者负责任地审查代码，使用什么工具不重要。代码质量才是关键。

2. **政策无效论**：这类政策只能限制诚实贡献者，无法阻止垃圾信息发送者。

3. **拒绝所有AI代码**：简单粗暴地拒绝所有AI代码可以节省时间。

4. **适应性而非限制**：不应人为限制，而应更多强调审查和批准所需的关注点。

## 讨论中的具体议题

- 19,000行的PR本身过大，无论是否使用AI都应被拒绝
- Node.js因此流失给Bun等竞争对手
- Bun已经在使用AI生成的提交（robobun），其代码质量令人担忧
- Elixir社区因AI最大化主义者而失去持不同意见的开发者（幸存者偏差）

## 结论

社区对是否应禁止AI辅助代码存在分歧。一方认为AI代码会带来质量风险和维护负担，另一方则认为应该关注代码质量而非工具，政策只会限制诚实贡献者而无法阻止垃圾信息。
