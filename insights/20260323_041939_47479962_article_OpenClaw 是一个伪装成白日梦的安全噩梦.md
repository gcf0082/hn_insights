## 概述

OpenClaw是一款基于AI的自主助手，能够访问用户的本地文件系统、终端、浏览器、电子邮件(Gmail)、Slack以及家庭自动化系统等功能。然而，这篇文章深入分析了OpenClaw在安全方面的严重缺陷，包括供应链漏洞、提示词注入风险、身份验证滥用以及大规模暴露的部署实例等问题。

## 主要安全问题

### 1. ClawdHub技能商店的恶意软件风险

OpenClaw依赖Skills(技能)扩展功能，用户可以从ClawdHub下载第三方上传的技能。然而，安全研究人员Jason Melier(1Password)发现，下载量最高的"Twitter"技能实际上是一个恶意软件投递向量：

- 该技能诱导用户安装名为"openclaw-core"的"必需依赖项"
- 提供的链接指向恶意基础设施
- 攻击流程包括：诱导运行命令 → 解码混淆的有效载荷 → 获取第二阶段脚本 → 下载并执行恶意二进制文件

Snyk对3,984个技能的security分析发现，约7.1%(283个)存在严重安全漏洞，会以明文形式在LLM上下文窗口中暴露敏感凭证。

### 2. 永久性提示词注入威胁

提示词注入是LLM固有的问题，但OpenClaw的多渠道特性(WhatsApp、Telegram、邮件)大大扩大了攻击面。任何人都可以向用户发送消息或嵌入指令来攻击AI助手。Gary Marcus指出，这些系统"以用户身份"运作，越过了操作系统和浏览器提供的安全保护，包括应用隔离和同源策略都不适用。

### 3. 认证滥用与过度授权令牌

OpenClaw需要存储OAuth访问令牌和刷新令牌来完成集成功能，但这些令牌以文件形式存储在本地。如果攻击者获得OpenClaw实例的访问权限，这些令牌将成为攻击目标。许多部署为了便利性而采用弱认证配置，使得"互联网暴露"到"令牌窃取"的路径非常短暂。

### 4. 内存作为Markdown文件的安全隐患

OpenClaw的内存系统完全由Markdown文件组成。被入侵的AI可以重写自己的内存文件，在用户毫不知情的情况下执行任务、静默窃取个人数据和凭证。

### 5. 大规模暴露的实例

安全公司Censys在2026年1月27-31日期间发现了约21,000个暴露的OpenClaw实例。BitSight在1月27日至2月8日的扫描中发现了超过30,000个存在漏洞的OpenClaw/Clawdbot/Moltbot实例。初始版本存在关键漏洞：来自localhost的流量被自动视为合法请求，导致任何人都可以绕过认证。

## 缓解建议

### 容器化部署

- 使用Docker容器隔离运行，避免在主机上直接部署
- 不挂载完整主目录，仅提供单一工作目录
- 使用非root用户运行容器，启用只读文件系统
- 禁止挂载Docker socket

### 云VPS部署

- 将网关绑定到127.0.0.1，仅通过VPN或私有隧道访问
- 配置防火墙，仅允许来自可信IP的SSH访问
- 使用rootless Docker模式
- 制定令牌轮换计划

### 账户隔离

为OpenClaw创建独立的Gmail、日历和各平台账户，而不是授予其访问个人主要账户的权限。

### 最小权限原则

- 根据风险级别分离不同的AI代理
- 对提升的访问权限设置时间限制
- 仅授予特定资源而非全部资源的访问权限
- 将破坏性操作设置为需要人工审批

## OWASP AI Agent十大风险映射

| OWASP Agent风险 | OpenClaw实现 |
|----------------|-------------|
| A01: 提示词注入(直接/间接) | 网页搜索结果、消息、第三方技能注入指令 |
| A02: 不安全的代理工具调用 | 基于未信任内存源的推理调用工具(bash、文件IO、邮件等) |
| A03: 过度代理自主性 | 单代理拥有文件系统根访问、凭证访问和网络通信权限 |
| A04: 缺少人工介入控制 | 即使受旧的不信任内存影响，破坏性操作也无需审批 |
| A05: 代理内存污染 | 所有内存来源无区分存储，无信任级别或过期机制 |
| A06: 不安全的第三方集成 | 第三方技能以完整代理权限运行，可直接写入持久内存 |
| A07: 权限分离不足 | 单代理处理不受信任的输入接收和高权限操作执行 |
| A08: 供应链模型风险 | 使用上游LLM但未验证微调数据或安全对齐 |
| A09: 无界代理间操作 | 当前为单体代理，未来多代理版本可能存在无约束通信 |
| A10: 缺少运行时监控与防护 | 内存检索→推理→工具调用之间无策略执行层 |

## 结论

尽管OpenClaw提供了强大的AI助手功能，但其安全风险远超普通软件工具的特性。在AI技术仍处于发展阶段的当下，必须谨慎处理并设置合理的防护栏：假设AI会被操纵，设计能够遏制和恢复错误的系统。对于普通用户，建议避免使用OpenClaw等高风险工具；对于必须使用的用户，应严格遵循容器化部署、账户隔离和最小权限等安全最佳实践。
