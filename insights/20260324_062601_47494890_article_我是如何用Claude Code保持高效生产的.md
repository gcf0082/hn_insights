## 内容摘要

本文作者 Neil Kakkar 分享了他在加入 Tano 公司六周后，如何利用 Claude Code 显著提升工作效率的经验。通过一系列基础设施优化和工作流程改进，他将传统的"独自开发"模式转变为"管理 AI 代理团队"的模式，代码提交量大幅增加。

## 核心要点

### 1. 自动化繁琐工作

作者意识到手动创建 Pull Request 是毫无价值的重复劳动。他编写了第一个 Claude Code 技能 `/git-pr`，自动化了从暂存更改、编写提交信息、制作 PR 描述到推送到 GitHub 的整个流程。有趣的是，AI 生成的 PR 描述比他自己写的更详细，因为它会读取完整的 diff 并正确总结更改。

**关键收获**：真正的改变是心态上的——从"代码实现者"转变为"AI 代理的管理者"。

### 2. 消除等待时间

预览更改需要停止当前工作、关闭开发服务器、在新分支上重启、检查预览。作者将构建系统切换到 SWC，服务器重启时间从约一分钟缩短到不到一秒。这意味着永远不会离开心流状态——保存文件，服务器已经启动，检查预览。注意力不会有任何漂移的间隙。

### 3. 让 Claude 看到实际效果

之前作者需要手动检查每个 UI 更改，亲眼验证是否符合预期。通过 Claude Code 的预览功能，代理可以设置预览、保存会话数据、查看 UI 实际外观。作者将这个功能融入工作流程：更改只有在代理验证了 UI 本身后才算"完成"。这意味着他只需进行最终审核，代理能自行发现错误。

### 4. 并行处理一切

快速重建和自动化预览暴露了另一个摩擦点：作者一次只能舒适地处理一个任务。他建立了一个系统，为每个 worktree 分配唯一范围的端口，避免冲突。这使他能够同时运行五个 worktree，创建循环变为：在不同的 worktree 上同时启动多个代理，每个代理构建不同的功能。

### 5. 关键在于基础设施，而非 AI

作者的角色发生了根本性变化。他仍然会从解决复杂问题和精心制作 UI 中获得乐趣，但少多了。更重要的是，他开始享受构建让代理更有效的 infrastructure——管理一个"十人团队"而不是独自开发。最好的成果不是写功能，而是构建让代码提交量从涓涓细流变成洪流的基础设施。

## 工作流程循环

每个阶段消除了一种不同类型的摩擦：

1. `/git-pr` 消除了**格式化**的摩擦——将代码更改转换为可展示的 PR
2. SWC 消除了**等待**的摩擦——做出更改和看到结果之间的死时间
3. 预览功能消除了**验证更改**的摩擦——快速查看正在发生什么
4. worktree 系统消除了**上下文切换**的摩擦——同时处理多项工作而不冲突

消除一个摩擦后，下一个变得可见。这是经典的约束理论——修复一个，系统立即向你展示下一个。

## 结论

当反馈循环足够紧密时，工程成为一种娱乐方式。游戏变成了：我能以多快的速度前进？当循环足够紧密，改进速度本身就是一种乐趣。
