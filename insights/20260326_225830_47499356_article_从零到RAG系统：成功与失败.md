## 摘要

本文作者分享了从零构建一个企业内部RAG（检索增强生成）系统的完整经历。该系统用于帮助工程师通过自然语言查询公司近十年来的所有项目文档，包括OrcaFlex文件等专业技术文档。最终成功构建了一个基于本地LLM的对话系统，处理了451GB的文档数据。

---

## 问题一：技术选型

**挑战**：需要本地运行的LLM（出于保密要求），支持自然语言问答，能快速响应。

**解决方案**：
- **LLM**：选择Ollama运行LLaMA模型（使用llama3.2:3b）
- ** Embeddings**：使用nomic-embed-text，性能优秀且适合技术文档
- **RAG框架**：LlamaIndex，负责文档索引、向量生成和查询
- **编程语言**：Python（Ollama和LlamaIndex都有优秀的Python SDK）

---

## 问题二：文档混乱

**挑战**：数据源是Azure上的一个文件夹，包含数百GB、数千个文件，格式多样，没有任何组织结构。

**问题**：首次运行RAG时，LlamaIndex尝试处理所有文件（包括视频、模拟文件、备份文件等），导致内存溢出，系统崩溃。

**解决方案**：添加过滤器，排除以下类型的文件：
- 视频：mp4, avi, mov, mkv等
- 图片：jpg, png, gif等
- 可执行文件：exe, dll等
- 压缩文件：zip, rar, 7z等
- 模拟文件：sim, dat
- 临时文件：tmp, temp, cache, log等
- 备份文件：bak, old等
- 邮件文件：msg, pst, eml等

同时排除CSV、JSON等价值低但处理成本高的文件。

**结果**：文件数量减少54%，内存问题解决。

---

## 问题三：索引451GB文档

**挑战**：使用LlamaIndex默认的JSON格式存储向量索引，在处理大规模文档时不可行：
- 服务重启需从头开始处理所有文档（耗时数天）
- JSON格式不适合大规模搜索

**解决方案**：采用ChromaDB向量数据库（基于SQLite）：
- 批处理pipeline，每次处理150个文件
- 生成向量后直接存储到ChromaDB
- 支持checkpoint（断点续索引）
- 易于备份和恢复

**结果**：738,470个向量，54GB索引，可以跨多个会话完成索引。

---

## 问题四：GPU性能不足

**挑战**：笔记本集成显卡处理500MB文档需4-5小时，效率极低。

**解决方案**：
- 租用配备NVIDIA RTX 400 SFF Ada（20GB VRAM）的虚拟机
- 修改容器以利用GPU加速
- 索引耗时2-3周完成

**成本**：184欧元（Hetzner云服务）。

---

## 问题五：用户体验

**解决方案**：
- **后端API**：Flask + Gunicorn
- **前端界面**：Streamlit（原生支持问答Widget）
- **响应格式**：每个回答显示信息来源和原始文档链接

---

## 问题六：文档存储问题

**挑战**：生产环境VM磁盘空间有限（100GB），无法存储全部原始文档。

**解决方案**：原始文档保留在Azure Blob Storage中，RAG系统只存储向量索引（54GB）。用户点击回答中的链接时，通过SAS令牌直接从云端下载文档。

## 最终架构

| 层次 | 技术 | 用途 |
|------|------|------|
| LLM | Ollama + llama3.2:3b | 本地响应生成 |
| Embeddings | nomic-embed-text | 文档向量化 |
| 向量数据库 | ChromaDB (HNSW) | 相似性存储和搜索 |
| RAG框架 | LlamaIndex | RAG流程编排 |
| API | Flask + Gunicorn | HTTP REST服务 |
| Web UI | Streamlit | 对话界面 |
| 容器 | Docker Compose | 服务编排 |
| GPU | NVIDIA Container Toolkit | 硬件加速 |
| 存储服务 | Azure Blob Storage | 云端持久化 |

## 经验教训

1. **内存管理**：不要一次性加载所有文档。使用批处理（作者使用150个/批），每批处理完后显式调用垃圾回收器释放内存。

2. **错误容忍**：即使有过滤器，仍可能遇到损坏的PDF、Word文档等。策略是记录失败的文件并继续处理，问题文件不应阻止整个批次。

3. **检查点系统**：每个批次可能耗时数小时，停机或重启不能从头开始。实现checkpoint系统保存最后完成的批次、节点数和时间戳。

4. **监控**：添加各类监控脚本：index-progress、index-watch、index-speed、index-checkpoint、index-failed等。

## 结论

系统并非完美，但足够使用。遗憾是无法让LLM直接运行OrcaFlex项目进行模拟，但最终结果令人满意：快速、可靠、对同事非常有用。

**建议**：如果你准备构建类似系统，花时间准备高质量的源数据。如果数据不够相关，LLM无法生成好的答案。
