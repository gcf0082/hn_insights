## 概述

`.claude` 文件夹是 Claude Code 的控制中心，存放着指令、自定义命令、权限规则和跨会话记忆。理解其结构可以让你配置 Claude Code 完全符合团队需求。

## 两个 .claude 目录

实际上存在两个 `.claude` 目录：

1. **项目级目录** (`项目根目录/.claude/`)：存放团队配置，可提交到 Git，团队成员共享相同的规则、命令和权限策略
2. **全局目录** (`~/.claude/`)：存放个人偏好和机器本地状态，如会话历史和自动记忆

## CLAUDE.md：Claude 的指令手册

这是整个系统中最重要的文件。当启动 Claude Code 会话时，它首先读取 `CLAUDE.md`，并在整个对话过程中记住其中的内容。

### 应当写入的内容

- 构建、测试和 lint 命令（如 `npm run test`、`make build`）
- 关键架构决策（如"我们使用 Turborepo 的 monorepo"）
- 不明显的陷阱（如"TypeScript 严格模式已启用，未使用变量会报错"）
- 导入约定、命名模式、错误处理风格
- 主要模块的文件和文件夹结构

### 不应写入的内容

- 属于 linter 或 formatter 配置的内容
- 可以直接链接到的完整文档
- 解释理论的冗长段落

建议将 `CLAUDE.md` 保持在 200 行以下，超过这个长度会占用过多上下文，Claude 的指令遵循能力反而会下降。

## CLAUDE.local.md：个人覆盖配置

如果有些偏好是你个人的、而非整个团队的，可以创建 `CLAUDE.local.md`。Claude 会与主 `CLAUDE.md` 一起读取它，且会自动被 Git 忽略。

## rules/ 文件夹：可扩展的模块化指令

当团队成长时，`CLAUDE.md` 会变得难以维护。`rules/` 文件夹解决了这个问题。

`.claude/rules/` 中的每个 markdown 文件都会与 `CLAUDE.md` 自动一起加载。可以按关注点拆分指令：

```
.claude/rules/
├── code-style.md
├── testing.md
├── api-conventions.md
└── security.md
```

### 路径作用域规则

通过 YAML frontmatter 可以让规则只在处理匹配文件时激活：

```yaml
---
paths:
  - "src/api/**/*.ts"
  - "src/handlers/**/*.ts"
---
# API 设计规则
```

Claude 处理不相关文件时不会加载该规则。

## commands/ 文件夹：自定义斜杠命令

`.claude/commands/` 中的每个 markdown 文件都会变成一个斜杠命令。文件名为命令名：`review.md` 创建 `/project:review`。

### 向命令传递参数

使用 `$ARGUMENTS` 传递命令名后的文本：

运行 `/project:fix-issue 234` 会将 issue 234 的内容直接注入提示词。

### 个人命令 vs 项目命令

项目命令在 `.claude/commands/` 中，会提交并与团队共享。个人命令放在 `~/.claude/commands/` 中，显示为 `/user:command-name`。

## skills/ 文件夹：按需调用的可重用工作流

Skills 与 Commands 的区别在于触发方式：

- **Commands**：等待你输入斜杠命令
- **Skills**：监控对话，当任务匹配技能描述时自动调用

每个 skill 存放在自己的子目录中，包含 `SKILL.md` 文件：

```
.claude/skills/
├── security-review/
│   ├── SKILL.md
│   └── DETAILED_GUIDE.md
└── deploy/
    ├── SKILL.md
    └── templates/
        └── release-notes.md
```

`SKILL.md` 使用 YAML frontmatter 描述何时使用：

```yaml
---
name: security-review
description: 全面的安全审计。用于审查代码漏洞、部署前或用户提及安全时
allowed-tools: Read, Grep, Glob
---
```

个人 skills 放在 `~/.claude/skills/`。

## agents/ 文件夹：专业子代理人格

当任务复杂到需要专门的专家时，可以在 `.claude/agents/` 中定义子代理人格。每个 agent 是带有自己系统提示词、工具访问权限和模型偏好的 markdown 文件：

```yaml
---
name: code-reviewer
description: 专业代码审查员。主动用于审查 PR、检查 bug 或验证实现
model: sonnet
tools: Read, Grep, Glob
---
```

- **tools 字段**：限制 agent 能做什么，安全审计员只需要 Read、Grep、Glob，不应该写文件
- **model 字段**：允许使用更便宜、更快的模型处理专注任务

个人 agents 放在 `~/.claude/agents/`。

## settings.json：权限和项目配置

`settings.json` 控制 Claude 允许和禁止做什么：

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Read",
      "Write",
      "Edit"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(curl *)",
      "Read(./.env)",
      "Read(./.env.*)"
    ]
  }
}
```

- **allow 列表**：无需确认即可运行的命令
- **deny 列表**：完全禁止的命令
- **中间地带**：未列入任一列表的命令会询问确认

也可以使用 `settings.local.json` 进行个人覆盖（自动 gitignored）。

## 全局 ~/.claude/ 文件夹

- `~/.claude/CLAUDE.md`：加载到每个 Claude Code 会话的个人编码原则和偏好
- `~/.claude/projects/`：存储每个项目的会话转录和自动记忆
- `~/.claude/commands/`、`~/.claude/skills/`、`~/.claude/agents/`：跨所有项目可用的个人命令、技能和代理

## 实践设置步骤

1. 运行 `/init` 生成起始 `CLAUDE.md`
2. 添加 `.claude/settings.json` 配置 allow/deny 规则
3. 创建一两个最常用工作流的命令
4. 当 `CLAUDE.md` 变拥挤时，拆分为 `.claude/rules/` 文件
5. 添加 `~/.claude/CLAUDE.md` 存放个人偏好

## 核心要点

`.claude` 文件夹本质上是一个协议，用于告诉 Claude 你是谁、你的项目做什么、应该遵循什么规则。定义得越清晰，纠正 Claude 的时间就越少，它做有用工作的时间就越多。

`CLAUDE.md` 是杠杆效应最高的文件。先把它做好，其他都是优化。
