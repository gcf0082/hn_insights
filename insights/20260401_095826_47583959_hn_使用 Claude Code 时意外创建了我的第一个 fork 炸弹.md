## 故事摘要

本文作者分享了使用Claude Code时意外创建"fork bomb"（叉式炸弹）的经历。作者编写了一个SessionStart钩子，试图在会话开始时捕获之前未触发的SessionEnd钩子，但由于设计缺陷，该钩子会触发两个新的Claude Code实例，而每个实例又会触发相同的钩子，导致指数级增长的进程创建，最终耗尽系统资源。

5. **fork bomb的技术解释** (connicpu): fork bomb不限于Unix的fork，任何递归生成多个子进程的方式都可称为fork bomb。

7. **作者的回应** (offbyone42): 原本是想创建一个会话摘要机制，但实现方式导致了fork bomb。现在已改用批处理作业方式。

8. **使用Git worktrees和VM** (MarcelOlsz): 建议使用Git worktrees和轻量级VM环境来安全地运行多个Claude Code会话。

## 关键教训

- **AI工具的意外后果**: 即使是经验丰富的开发者，也可能因对AI工具机制理解不足而引发意外问题
- **钩子函数的风险**: 自动触发的钩子可能带来指数级增长的风险
- **批处理优于即时触发**: 对于后台任务，批处理作业比即时触发更可靠、更安全
