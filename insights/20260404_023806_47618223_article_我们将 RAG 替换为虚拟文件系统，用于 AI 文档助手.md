## 核心摘要

Mintlify 团队为其文档助手构建了一个虚拟文件系统 **ChromaFs**，解决了传统沙箱方案的高延迟和高成本问题。该方案将现有的 Chroma 数据库作为虚拟文件系统的基础，将 UNIX 命令（grep、cat、ls、find）翻译为数据库查询，实现了即时启动（~100毫秒）和零边际计算成本。

---

## 问题背景

### RAG 的局限性

传统的 RAG（检索增强生成）仅能返回与查询匹配的文本片段。当答案跨越多个页面，或用户需要精确语法但该内容未进入 Top-K 结果时，系统将无法工作。

### 传统沙箱方案的痛点

1. **高延迟**: 使用 Daytona 等沙箱服务创建会话需要约 **46秒**（P90），包括 GitHub 克隆和其他设置
2. **高成本**: 按 850,000 次对话/月计算，即使是 minimal 配置（1 vCPU, 2 GiB RAM, 5分钟会话），年度计算成本也将超过 **70,000美元**

---

## ChromaFs 解决方案

### 架构设计

ChromaFs 基于 Vercel Labs 的 [just-bash](https://github.com/vercel-labs/just-bash) 构建——一个用 TypeScript 重新实现的 bash，支持 grep、cat、ls、find 和 cd 命令。ChromaFs 实现了 `IFileSystem` 接口，将所有文件系统调用转换为 Chroma 数据库查询。

### 核心指标对比

| 指标 | 传统沙箱 | ChromaFs |
|------|----------|----------|
| P90 启动时间 | ~46秒 | ~100毫秒 |
| 边际计算成本 | ~$0.0137/对话 | $0（复用现有基础设施） |
| 搜索机制 | 线性磁盘扫描（系统调用） | 数据库元数据查询 |
| 基础设施 | Daytona 或类似供应商 | 已配置的数据库 |

### 关键技术实现

#### 1. 目录树引导（Bootstrapping）

整个文件树以 gzip 压缩的 JSON 文档（`__path_tree__`）存储在 Chroma 集合中：

```json
{
  "auth/oauth": { "isPublic": true, "groups": [] },
  "auth/api-keys": { "isPublic": true, "groups": [] },
  "internal/billing": { "isPublic": false, "groups": ["admin", "billing"] },
  "api-reference/endpoints/users": { "isPublic": true, "groups": [] }
}
```

初始化时，服务器获取并解压文档到两个内存结构：
- `Set<string>`: 文件路径集合
- `Map<string, string[]>`: 目录到子项的映射

后续的 `ls`、`cd`、`find` 操作均在本地内存中解析，无需网络调用。

#### 2. 访问控制

ChromaFs 在构建文件树前使用用户会话令牌剪枝 slug，并将匹配过滤器应用于所有后续 Chroma 查询。如果用户无权访问某文件，该文件将从树中完全排除，代理无法访问甚至引用被剪枝的路径。

相比之下，传统沙箱需要管理 Linux 用户组、chmod 权限或维护每个客户层的隔离容器镜像。

#### 3. 页面重组（从分块重建）

Chroma 中的页面被拆分为分块用于嵌入。当代理运行 `cat /auth/oauth.mdx` 时，ChromaFs 获取所有匹配 slug 的分块，按 `chunk_index` 排序并连接成完整页面。结果被缓存，因此 grep 工作流中的重复读取不会再次访问数据库。

对于存储在客户 S3 桶中的大型 OpenAPI 规范，ChromaFs 注册了惰性文件指针，按需解析。

#### 4. 写操作处理

所有写操作都会抛出 `EROFS`（只读文件系统）错误。代理可以自由探索，但无法修改文档，这使系统无状态，无需会话清理，也无需担心一个代理破坏另一个代理的视图。

## Grep 优化

### 两阶段过滤策略

1. **粗筛（Chroma）**: 将 grep 标志解析为 Chroma 查询（`$contains` 用于固定字符串，`$regex` 用于模式），识别可能包含匹配项的文件
2. **精筛（内存）**: 将匹配的分块批量预取到 Redis 缓存，然后将 grep 命令重写为仅针对匹配文件，在内存中执行细过滤

这使得大型递归查询在毫秒内完成。

## 生产环境成果

ChromaFs 为 Mintlify 的文档助手提供支持，服务于 **数十万用户**，每天处理 **30,000+ 次对话**。

### 主要收益

- 即时会话创建（P90 从 46秒降至 100毫秒）
- 零边际计算成本（复用现有 Chroma 基础设施）
- 内置 RBAC（无需额外基础设施）

---

## 结论

ChromaFs 展示了如何通过将现有 RAG 基础设施重新利用为虚拟文件系统来解决 AI 助手的延迟和成本问题。这种方法不仅提升了用户体验，还显著降低了基础设施成本，为类似场景提供了有价值的参考架构。
