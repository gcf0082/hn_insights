## 漏洞概述

这是一个关于 OpenClaw 软件的权限提升漏洞（CVE-2026-33579）。OpenClaw 创作者在评论中解释称，这并非"任何随机 Telegram/Discord 消息都能立即控制所有 OpenClaw 实例"的情况。

### 根本原因

问题的根源在于之前的安全修复不完整。早期公告通过将调用者的作用域传递到核心批准检查来加固网关 RPC 路径的设备审批，但 `/pair approve` 插件命令路径仍然调用相同的批准函数而没有 `callerScopes` 参数，核心逻辑在该参数缺失时默认开放。

### 利用条件

最确认的利用路径是：已经拥有网关访问权限且有足够权限发送命令的客户端，可以使用 `chat.send` 配合 `/pair approve latest` 来批准请求更广泛作用域（包括 `operator.admin`）的待处理设备请求。这本质上是从配对/写入级别访问权限到管理员权限的作用域天花板绕过。

### 影响范围

漏洞并非仅针对 Telegram 或特定消息提供商，而是存在于共享的插件命令处理程序中。任何已经获得授权的命令发送者只要能访问 `/pair approve` 就可以利用该漏洞。

## 社区反应

### 创作者回应

OpenClaw 创作者表示，实际风险非常低，特别是当 OpenClaw 作为单用户个人助理使用时。他们正在与来自 Nvidia、ByteDance、Tencent 和 OpenAI 的开发者合作加强代码库。

### 争议焦点

1. **数据真实性**：有评论指出 135k 公开实例和 63% 零认证的数据缺乏可信来源
2. **代码质量**：有评论批评 OpenClaw 是 50 万+ 行的 vibe coding 产物，难以理解和推理
3. **默认配置**：有评论指出此前默认绑定到 0.0.0.0，可能导致用户无意间暴露于互联网

### 实际用途讨论

社区讨论了 OpenClaw 的实际用途：
- 抓取市政会议信息
- 自动预约健身房位置
- 作为其他代理的编排器
- 读取邮件并发送通知

## 安全建议

1. 确保 OpenClaw 实例未直接暴露在公网
2. 启用并配置身份验证
3. 关注官方安全公告和 CVE 披露
4. 考虑使用网络隔离或虚拟机部署
