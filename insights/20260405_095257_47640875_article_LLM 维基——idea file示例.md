# LLM Wiki：使用大语言模型构建个人知识库的模式

## 核心思想

当前大多数人与大语言模型交互文档的方式是 RAG（检索增强生成）：上传文件集合，LLM 在查询时检索相关片段并生成答案。这种方式存在一个问题：LLM 每次回答问题时都需要从头发现知识，没有任何积累。问一个需要综合五份文档的细微问题，LLM 必须每次都寻找并拼接相关碎片，没有任何东西是建立起来的。

**LLM Wiki 的想法不同**：LLM 不是在查询时仅从原始文档检索，而是在你和原始来源之间构建和维护一个持久的维基百科——一个结构化的、相互链接的 Markdown 文件集合。当你添加新来源时，LLM 不仅索引它供以后检索，而是读取它，提取关键信息，并将其整合到现有维基中——更新实体页面、修订主题摘要、标记新数据与旧声明相矛盾的地方、加强或挑战不断发展的综合。知识被编译一次，然后保持最新，而不是每次查询时重新推导。

**关键区别在于：维基是一个持久的、复合的产物。** 交叉引用已经存在，矛盾已经被标记，综合已经反映了你所阅读的一切。维基随着你添加的每个来源和提出的每个问题而变得越来越丰富。

你永远不会（或很少）自己写维基——LLM 写和维护所有这些。你负责来源、探索和提出正确的问题。LLM 做所有繁琐的工作——总结、交叉引用、归档和记账，使知识库随着时间的推移真正有用。

## 架构

架构包含三层：

1. **原始来源** - 你策划的来源文档集合。文章、论文、图片、数据文件。这些是不可变的——LLM 从中读取但从不修改它们。这是你的真理来源。

2. **维基** - LLM 生成的 Markdown 文件目录。包括摘要、实体页面、概念页面、比较、概述和综合。LLM 完全拥有这一层。它创建页面、在新来源到达时更新它们、维护交叉引用并保持一切一致。你阅读它；LLM 写它。

3. **模式/大纲** - 一个文档（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告诉 LLM 维基的结构方式、约定是什么，以及在摄取来源、回答问题或维护维基时遵循的工作流程。这是关键配置文件——它使 LLM 成为有纪律的维基维护者而不是通用聊天机器人。你和 LLM 随着时间的推移共同发展这一层，因为你弄清楚什么适合你的领域。

## 操作

### 摄取（Ingest）

你将新来源放入原始集合中，并告诉 LLM 处理它。一个示例流程：LLM 阅读来源与你讨论关键要点、在维基中写摘要页面、更新索引、更新维基中相关的实体和概念页面，并将条目附加到日志中。单个来源可能涉及 10-15 个维基页面。

### 查询（Query）

你针对维基提出问题。LLM 搜索相关页面，阅读它们，并综合答案并附上引用。答案可以根据问题采取不同的形式——Markdown 页面、比较表、幻灯片（Marp）、图表（matplotlib）、画布。重要的洞察：**好的答案可以作为新页面存回维基。** 你要求的比较、分析、你发现的连接——这些都是有价值的，不应该消失在聊天历史中。这样，你的探索就像摄取的来源一样在知识库中复合。

### 清理（Lint）

定期要求 LLM 对维基进行健康检查。寻找：页面之间的矛盾、被新来源取代的过时声明、没有入站链接的孤立页面、被提及但缺乏自己页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的数据空白。LLM 善于建议新问题来调查和新来源来寻找。这随着维基的增长保持维基的健康。

## 索引和日志

两个特殊文件帮助 LLM（和你）驾驭不断增长的维基：

- **index.md** 是面向内容的。它是维基中所有内容的目录——每个页面列出并带有链接、一行摘要以及可选的元数据（如日期或来源数量）。按类别组织（实体、概念、来源等）。LLM 在每次摄取时更新它。回答问题时，LLM 首先读取索引以找到相关页面，然后深入研究它们。这在中等规模（约 100 个来源、数百个页面）下出奇地有效，避免了需要基于嵌入的 RAG 基础设施。

- **log.md** 是按时间顺序的。它是发生了什么以及何时发生的append-only记录。一个有用的提示：如果每个条目以一致的前缀开头（例如 `## [2026-04-02] ingest | Article Title`），日志可以用简单的 unix 工具解析——`grep "^## \[" log.md | tail -5` 给你最后 5 个条目。日志让你看到维基的演变时间线，并帮助 LLM 理解最近做了什么。

## 为什么这有效

维护知识库的繁琐部分不是阅读或思考——而是记账。更新交叉引用、保持摘要最新、标记新数据何时与旧声明相矛盾、维护数十个页面的一致性。人类放弃维基是因为维护负担增长得比价值快。LLM 不会无聊、不会忘记更新交叉引用、可以一次触碰 15 个文件。维基保持维护是因为维护成本接近于零。

人类的工作是策划来源、引导分析、提出好问题、思考这一切意味着什么。LLM 的工作是其他一切。

这个想法在精神上与 Vannevar Bush 的 Memex（1945 年）相关——一个个人策划的知识存储，文档之间具有关联路径。布什的愿景更接近于此，而不是网络变成的样子：私人、积极策划，文档之间的连接与文档本身一样有价值。他无法解决的部分是谁来做维护。LLM 处理这个问题。

## 社区反馈与实现

这个模式引发了广泛的讨论和多种实现：

- **engram** - 一个类似实现，代理读取来源、更新相关维基页面、标记与现有知识的矛盾
- **Obsidian Seed** - 一个发现驱动的向导，通过对话构建个性化的 Obsidian 保险库
- **Mnemon** - 知识提取管道，实现 Raw → Wiki → Frontend
- **Context Lattice** - 将 RAG、Graph、Vector 和语义服务结合的元 RAG 系统
- **Veritas Acta** - 多模型验证、带签收据的零信任验证系统
- **wiki-skills** - 通过技能文件实现这些想法

## 关键要点

1. **持久性**：维基是一个持续积累的产物，而不是每次查询时重新检索的文档集合
2. **自动化维护**：LLM 承担繁琐的记账工作，包括更新交叉引用和标记矛盾
3. **三层架构**：原始来源、LLM 生成的维基、模式/大纲配置文件
4. **索引和日志**：index.md 用于内容导航，log.md 用于时间线追踪
5. **查询到知识**：有价值的查询结果应存回维基，实现知识的复合增长
6. **真实性优先**：强调 LLM 应该是编辑而非创作者，每个主张都应追溯到来源
