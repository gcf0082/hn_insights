## 核心发现总结

UC Berkeley研究团队构建了一个自动化扫描代理，系统性地审计了八个最流行的AI Agent基准测试（SWE-bench、WebArena、OSWorld、GAIA、Terminal-Bench、FieldWorkArena、CAR-bench），发现了令人震惊的结论：**每个基准都可以被利用来获得接近完美的分数，而无需解决任何实际任务**。

这些不是理论攻击。研究团队的代理构建了实际可工作的漏洞，并通过了官方评估管道，证明了分数可以完全被操控。

## 技术细节

### SWE-bench：100%解决率，无需修复任何bug

- **漏洞**：在代理的补丁应用容器中，任何引入的代码都在测试前执行。一个10行的`conftest.py`文件可以使用Pytest hooks强制所有测试报告为通过：

```python
import pytest
@pytest.hookimpl(hookwrapper=True)
def pytest_runtest_makereport(item, call):
    outcome = yield
    rep = outcome.get_result()
    if rep.when == "call":
        rep.outcome = "passed"
```

- **结果**：SWE-bench Verified的500个实例和Pro的731个实例都可以100%被利用。

### Terminal-Bench：100%分数，一个二进制木马

- **漏洞**：容器有完全的互联网访问。代理在验证阶段用木马替换`/usr/bin/curl`，当验证器运行`curl | sh`时，拦截请求并Trojanize `uvx`二进制，使其对pytest调用产生假的通过输出。
- **结果**：89个任务，100%分数，无需编写任何解决方案代码。

### WebArena：直接读取答案

- **漏洞**：任务配置包含黄金答案JSON文件。Playwright的Chromium可以导航到`file://` URL直接读取本地文件。
- **结果**：~100%在所有812个任务上。

### FieldWorkArena：任意消息即满分

- **漏洞**：验证函数`validate()`只检查最后一条消息是否来自助手，**完全忽略消息内容**。

```python
def validate(self, page, chat_messages):
    if chat_messages[-1]["role"] == "assistant":
        return 1.0, True, "Recieved answer", {}  # 任何答案 = 1.0
```

- **结果**：只需发送`{}`即可获得890个任务100%分数。

### OSWorld：下载黄金答案

- **漏洞**：VM有不受限制的互联网访问。黄金文件托管在公开的HuggingFace URL上。代理直接下载到评估器检查的路径。
- **结果**：完美匹配，评分1.0。

### GAIA：规范化攻击

- **漏洞**：验证答案是公开的。规范化函数剥离所有空白和标点符号，使任何字符串都可以通过匹配。
- **结果**：~98%分数。

### CAR-bench：LLM评判注入

- **漏洞**：代理内容直接插入LLM评判提示，未进行清理。简单的HTML注释可以偏置评判。
- **结果**：100% hallucination任务分数。

## 七大漏洞模式

1. **代理与评估器之间无隔离**：代理代码与评估器检查的环境共享。
2. **答案随测试一起分发**：参考答案嵌入在任务配置中。
3. **对不受信任的输入使用eval()**：启用任意代码执行。
4. **LLM评判无输入清理**：易于Prompt注入。
5. **字符串匹配过于宽松**：任何冗长答案都能通过。
6. **评估逻辑不评估**：验证函数不检查答案正确性。
7. **信任不受信任代码的输出**：评估基础设施可被妥协。

## 影响

- 模型选择：基于基准分数选择模型的团队可能在比较噪声。
- 投资：资助决策受可被操控的排行榜位置影响。
- 安全评估：能力基准可以被 inflation，安全基准可能同样脆弱。
- 研究方向：研究人员优化基准性能，如果基准被破坏，领域优化的是错误目标。

## 建议

研究团队提出了**Agent-Eval Checklist**：

- **隔离代理与评估器**：在单独环境中运行评估，不信任来自代理沙箱的文件或状态。
- **永不eval()不受信任的输入**：使用适当的解析器。
- **清理LLM评判输入**：使用结构化标记和输出格式。
- **对抗性测试评估器**：在发布前尝试打破基准，使用null代理和随机代理。
- **防止篡改评估数据**：将代理环境的所有工件视为不受信任。
- **使评分健壮**：避免子字符串匹配，不 silent 排除失败任务。
- **保密答案**：从不发布地面真相，定期轮换测试实例。

## 结论

基准驱动真实决策。当一个零能力的代理比复杂系统获得更高分数时，基准作为能力衡量标准失败了。

研究团队正在开发**BenchJack**，一个通用的Agent基准漏洞扫描器，作为基准开发生命周期的标准步骤。
