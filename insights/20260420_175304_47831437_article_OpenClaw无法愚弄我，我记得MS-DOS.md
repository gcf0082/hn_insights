## 文章摘要

本文探讨了构建安全本地AI代理的架构问题，作者以MS-DOS时代的安全教训为鉴，对比了NVIDIA NemoClaw教程与作者自研的Wirken项目在安全设计上的差异。

### 核心观点

1. **历史教训**: MS-DOS时期缺乏安全防护，任何程序都可以访问内核、挂钩中断、写入磁盘任意位置，导致Wal-Mart等公司在2006年遭受数据泄露，直到2009年才通报。

2. **当前问题**: AI Agent网关设计仿佛在走DOS的老路，存在单令_token、整个沙箱运行等安全隐患。

3. **NVIDIA方案**: NemoClaw教程通过以下步骤解决安全问题：
   - 将Ollama绑定到0.0.0.0使沙箱代理可跨网络命名空间访问推理服务
   - 通过Telegram聊天频道发送配对代码
   - 在主机侧TUI中批准被阻止的出站连接

4. **Wirken方案**: 作者的替代架构
   - 每个通道是独立进程，拥有独立Ed25519身份
   - 保险库在进程外运行
   - 推理保持在loopback（代理在主机上）
   - Shell执行在工具层配置的加固容器中运行

### 安全机制对比

| 步骤 | NemoClaw | Wirken |
|------|----------|-------|
| 运行时 | 注册NVIDIA容器运行时，设置cgroup命名空间模式 | 无等效步骤，仅作为shell exec的沙箱 |
| Ollama | 覆盖OLLAMA_HOST为0.0.0.0 | 保持在127.0.0.1 |
| 安装 | curl-pipe-bash | curl-pipe-sh，用ssh-keygen验证发布签名 |
| 模型 | ollama pull后预加载到GPU | 相同模式 |
| 入站 | 沙箱图像包含策略和推理 | 写入provider配置和通道注册 |
| Telegram | 聊天频道发送配对代码 | Bot token放入加密保险库 |
| Web UI | 本地URL带fragment中的capability token | loopback绑定，无需token |
| 远程访问 | 通过OpenShell端口转发+SSH隧道 | 仅SSH隧道 |
| 策略 | 在netns边界执行，批准后持续 | 在工具调度层执行，16个高风险命令前缀始终提示 |

## 主要结论

作者认为AI代理领域仍处于早期阶段，应借鉴计算机历史已有的安全经验。关键问题是是否将Unix自1973年以来就具备的进程分离、用户分离、文件权限等安全机制应用到AI代理中。
