# Axios 第三方开发者工具供应链攻击洞察

**洞察链接**: https://openai.com/index/axios-developer-tool-compromise/

**发布日期**: 2026年4月10日

**相关安全事件**: Axios npm供应链妥协

---

## 事件概述

2026年3月31日，广受欢迎的JavaScript HTTP客户端库Axios遭遇供应链攻击，被注入恶意依赖。这是同期广泛报道的软件供应链安全事件的一部分。OpenAI在发现其macOS应用签名流程受到影响后，迅速采取应对措施。

## 攻击技术分析

### 攻击手法

攻击者首先于2026年3月30日使用一个一次性ProtonMail邮箱发布了一个伪装的crypto-js库。随后在3月31日清晨，攻击者控制了Axios主要维护者的GitHub和npm账户，并将邮箱更改以阻止原所有者登录，随后发布了带有恶意代码的Axios版本1.14.1和0.30.4。

这些恶意版本包含`plain-crypto-js@4.2.1`依赖，这是一个虚假运行时依赖，通过安装后脚本自动执行，无需用户交互。当开发者运行`npm install axios@1.14.1`或`npm update`时，npm会自动解析依赖树并安装plain-crypto-js，然后执行其postinstall脚本，触发恶意dropper。

### 攻击 Payload

dropper会连接到一个已知恶意命令控制（C2）服务器，该服务器由North Korean state actor group Sapphire Sleet运营。服务器会根据目标系统平台（Windows、macOS、Linux）投递不同的第二阶段远程访问木马（RAT）。完成投递后，恶意软件会自毁并用干净的decoy替换package.json，使开发者事后检查node_modules文件夹时无法发现任何异常。

## OpenAI 受影响情况

OpenAI在公告中披露，其macOS应用签名流程中使用的GitHub Actions工作流在2026年3月31日下载并执行了恶意版本的Axios（1.14.1版本）。该工作流有权访问用于签署macOS应用程序（包括ChatGPT Desktop、Codex、Codex-cli和Atlas）的证书和公证材料。

OpenAI的内部分析得出结论，由于payload执行时机、证书注入到job的时序、job本身的顺序以及其他缓解因素，签名证书很可能未被成功窃取。尽管如此，出于高度谨慎的考虑，OpenAI仍将证书视为已泄露，并进行了撤销和轮换。

## 安全影响与应对措施

### 潜在影响

- **签名证书风险**: 暴露的证书可能被用于签署伪造的OpenAI macOS应用
- **用户数据**： OpenAI明确表示未发现用户数据被访问的证据
- **系统渗透**: 使用受影响Axios版本的系统可能已被植入RAT

### 缓解措施

OpenAI采取的应对措施包括：

1. 撤销并轮换所有受影响的签名证书
2. 使用新证书重新构建并发布所有macOS应用
3. 阻止使用受影响的公证材料进行任何进一步macOS应用公证
4. 建议macOS用户更新到最新版本的应用

CISA和Microsoft发布的安全指南建议：

1. 回滚到安全版本（axios@1.14.0或axios@0.30.3）
2. 从node_modules中删除plain-crypto-js
3. 轮换可能暴露的所有凭证（VCS令牌、CI/CD密钥、云密钥、npm令牌、SSH密钥）
4. 在.npmrc中设置`ignore-scripts=true`防止恶意脚本执行
5. 设置`min-release-age=7`仅安装发布超过7天的包

## 威胁归因

Microsoft威胁情报将此次攻击归因于Sapphire Sleet，这是一个与North Korean政府有关联的网络威胁行为者组织。该组织以针对性的供应链攻击着称，主要针对加密货币交易所和区块链公司。

## 安全建议

对于使用Axios的开发者和组织：

1. **立即检查**：审计CI/CD pipeline日志，查找npm install执行是否更新到受影响版本
2. **回滚版本**：将Axios回滚到1.14.0或0.30.3并固定版本号
3. **添加覆盖**：在package.json中添加overrides阻止恶意版本的传递依赖解析
4. **假设被入侵**：如果检测到plain-crypto-js，应假设系统已被入侵
5. **隔离调查**：将受影响系统从网络隔离并进行取证调查

## 总结

此次Axios供应链攻击展示了现代软件供应链的脆弱性。即使是广泛使用的开源库也可能在一夜之间被植入恶意代码。OpenAI的响应体现了成熟的 incident response 实践：即使分析表明证书可能未被窃取，仍将其视为已泄露并主动轮换，这种预防性安全措施对于维护用户信任至关重要。
