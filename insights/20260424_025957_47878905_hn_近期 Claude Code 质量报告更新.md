## 问题概述

Anthropic 就近期 Claude Code 质量下降问题发布了详细的事后分析报告。经调查，问题根源追溯到三个独立的代码变更，这些变更分别影响了 Claude Code、Claude Agent SDK 和 Claude Cowork，但并未波及 API 服务。所有三个问题已于 4 月 20 日（v2.1.116 版本）修复完成。

## 三个根本原因详解

### 1. 推理努力默认值调整（2025年3月4日）

**问题**：Anthropic 将 Claude Code 的默认推理努力从 `high`（高）调整为 `medium`（中），旨在减少用户在高模式下遇到的长时间等待问题。然而，这一调整导致了明显的智能降级，用户普遍反映 Claude 变得不够智能。

**修复**：该变更于 4 月 7 日回滚。所有用户现默认为 Opus 4.7 使用 `xhigh`  effort，其他模型使用 `high` effort。

### 2. 缓存优化引发的记忆丢失（2025年3月26日）

**问题**：一个本意为提升效率的缓存优化引入了严重 bug。原设计是当会话闲置超过一小时后清除旧的推理记录以降低延迟，但实现中的错误导致系统在会话的每一轮都清除推理历史，而非仅执行一次。这使得 Claude 表现出"健忘"和重复行为的特征。

**影响范围**：Sonnet 4.6 和 Opus 4.6  
**修复时间**：4 月 10 日（v2.1.101 版本）

### 3. 系统提示词变更导致输出质量下降（2025年4月16日）

**问题**：为了减少冗长输出，Anthropic 在系统提示词中添加了限制长度的指令。然而，结合其他提示词变更，这一改动损害了代码编写质量。

**具体指令**：
> "Length limits: keep text between tool calls to ≤25 words. Keep final responses to ≤100 words unless the task requires more detail."

**影响范围**：Sonnet 4.6、Opus 4.6 和 Opus 4.7  
**修复时间**：4 月 20 日

## 为何问题发现延迟

由于三个变更各自在不同时间和不同流量切片上生效，叠加效果呈现出广泛且不一致的退化特征。加之早期用户反馈难以与正常波动区分，内部测试也未能复现这些问题，导致问题的发现和诊断被显著推迟。

Anthropic 承认这次事件不代表用户对 Claude Code 的期望体验，并宣布将为所有订阅用户重置使用限额。

## 后续改进措施

1. **内部测试流程优化**：确保更多内部员工使用与公开版本完全一致的 Claude Code
2. **系统提示词变更加强审核**：为每个系统提示词变更运行全面的按模型评估，进行消融实验以理解每行影响
3. **渐进式发布**：对可能影响智能的变更添加浸泡期和更广泛的评估套件
4. **代码审查工具增强**：扩大支持代码仓库作为代码审查的上下文
5. **用户反馈渠道**：通过 @ClaudeDevs 在 X 上解释产品决策和在 GitHub 上集中发布更新

## 关键教训

- 即使经过多轮人工和自动化代码审查、单元测试、端到端测试和内部测试，bug 仍可能存在于边界情况中
- 当问题仅在特定corner case（陈旧会话）发生时，测试覆盖往往不足
- 看似微小的用户体验权衡（如延迟vs智能）可能产生重大影响
- 用户反馈是发现这类问题的关键因素
