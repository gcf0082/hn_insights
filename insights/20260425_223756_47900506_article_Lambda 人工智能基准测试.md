## 项目概述

LamBench 是一个用于评估 AI 模型在**纯 lambda 演算**（Pure Lambda Calculus）编程能力的基准测试套件。该项目由 Victor Taelin 创建，通过 120 道纯 lambda 演算编程题来测试 AI 模型的核心编程能力。

### 核心设计理念

传统的编程基准测试通常依赖已有的编程语言和库（如 Python、JavaScript），而 LamBench 采用了极端的简化策略：所有数据必须用 lambda 演算直接编码，不依赖任何内置数据类型。这意味着 AI 模型必须从零开始构建完整的数据结构和算法，对计算思维和抽象能力提出了极高要求。

### 编程语言：Lamb

Lamb 是一种极简的 lambda 演算语言，仅包含两个语法构造：
- `λx.body` — lambda 抽象
- `f(x,y)` — 函数应用（是 `((f x) y)` 的语法糖）
- `@name` — 引用顶层定义（支持递归）

示例程序：
```
@true  = λt.λf.t
@false = λt.λf.f
@not   = λb.λt.λf.b(f,t)
@main  = @not(@false)
```

## 技术实现

### 评估流程

1. **题目加载**：每个 `.tsk` 文件包含自然语言问题描述和测试用例
2. **模型输入**：向模型提供问题描述和测试用例，要求模型输出一个 `.lam` 程序
3. **执行与验证**：将每个测试表达式追加到模型的程序后，使用 Lamb 解释器进行规范化，然后比较输出
4. **评分标准**：只有当**所有**测试用例都产生完全匹配的预期规范形式时，题目才视为通过

### Lamb 解释器

Lamb 解释器负责对 lambda 表达式进行规范化（normalization），这是评估的核心环节。规范化过程通过重复 beta 归约直到表达式不再变化为止。

### 结果生成

评估结果写入 `res/` 目录，每个模型一个带时间戳的文本文件。着陆页（`docs/index.html`）通过 `bun run build` 脚本解析所有结果文件生成。

### 运行基准测试

```bash
bun install
bun bench <provider/model>

# 示例
bun bench openai/gpt-5.5
bun bench anthropic/opus-4.7
bun bench google/gemini-3.1-pro-preview

# 带选项运行
bun bench openai/gpt-5.5 --filter algo_ --concurrency 8 --timeout 300
```

支持以下参数：
- `--filter <prefix>` — 过滤特定前缀的题目
- `--concurrency <n>` — 并发数
- `--timeout <seconds>` — 超时时间
- `--no-reasoning` — 禁用推理模式

## 关键洞察

### 1. AI 模型在 lambda 演算领域的分化明显

顶级模型（GPT-5.3 Codex、Opus 4.6）以 90% 的通过率领跑，但大多数模型在 50% 以下。这表明 lambda 演算编程是一个有效的"能力分水岭"——它不仅考验模型的编程能力，更考验模型对计算本质（而非具体语言语法）的理解深度。

### 2. 算法类题目是最具挑战性的

Brainfuck 解释器、旅行商问题（TSP）、数独求解器等算法题目的实现难度远高于基础数据结构操作。这些题目要求模型不仅理解 lambda 编码，还要能在纯 lambda 演算的约束下实现复杂的控制流和数据结构。

### 3. Church 编码与 Scott 编码的对称性设计

该基准测试巧妙地设计了对称的编码体系——Church 和 Scott 两种编码方式各有一套完整的数据结构题目。这种设计可以评估模型是否真正理解了 lambda 编码的通用原理，而非依赖特定的编码惯例。

### 4. v2 评分将引入程序大小指标

当前 v1 评分仅基于通过率。未来的 v2 版本计划将程序大小（以 bit 为单位与参考实现的比值）纳入评估，这意味着不仅要求正确性，还要求解决方案的优雅程度。

### 5. 纯 lambda 演算是 AI 编程能力的"极限测试"

通过移除所有高级语言特性和标准库，LamBench 将编程问题还原到最原始的形式。这种极端设置揭示了一个有趣的现象：即使是最强大的模型，在没有语法糖和库函数的情况下，完成复杂算法的难度显著上升。
