**作者**：Arpit Bhayani

**发布日期**：2025年

**标签**：数据库架构、人工智能 Agent、系统设计

---

## 文章摘要

本文探讨了传统数据库架构在面对 AI Agent 工作负载时面临的系统性挑战。文章指出，数据库的底层设计假设——调用者是确定性的应用程序、写操作经过人工审查、连接短暂、故障明显——正在被 Agentic AI 系统全面打破。

---

## 核心洞察

### 1. 确定性调用者假设的失效

传统系统中，SQL 查询由人类开发者编写、审查和测试。Agent 则通过推理生成查询，同一任务可能产生从未见过的复杂 JOIN 语句。

**应对策略**：
- 在角色级别设置语句超时（`statement_timeout`）
- 为空闲事务设置超时（`idle_in_transaction_session_timeout`），防止 Agent 暂停推理时长期占用连接

```sql
CREATE ROLE agent_worker;
ALTER ROLE agent_worker SET statement_timeout = '5s';
ALTER ROLE agent_worker SET idle_in_transaction_session_timeout = '10s';
```

### 2. 写操作意图假设的失效

Agent 可能基于错误理解自主执行写操作、在循环中重复写入、或因网络错误误判而重试。一次静默失败可能导致数千行错误数据的写入。

**应对策略**：

**软删除**：永远不让 Agent 执行硬删除操作。

```sql
ALTER TABLE orders ADD COLUMN deleted_at TIMESTAMPTZ;
ALTER TABLE orders ADD COLUMN deleted_by TEXT;
ALTER TABLE orders ADD COLUMN delete_reason TEXT;

CREATE VIEW active_orders AS
  SELECT * FROM orders WHERE deleted_at IS NULL;
```

**仅追加事件日志**：高风险操作采用事件溯源模式。

```sql
CREATE TABLE order_state_log (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  order_id UUID NOT NULL REFERENCES orders(id),
  previous_status TEXT,
  new_status TEXT NOT NULL,
  changed_by TEXT NOT NULL,
  changed_at TIMESTAMPTZ DEFAULT now(),
  reason TEXT,
  idempotency_key TEXT UNIQUE
);
```

**幂等性键**：Agent 重试是设计需求，幂等性键不可或缺。

```python
import hashlib

def make_idempotency_key(task_id: str, operation: str, target_id: str) -> str:
    raw = f"{task_id}:{operation}:{target_id}"
    return hashlib.sha256(raw.encode()).hexdigest()[:32]
```

### 3. 连接短暂假设的失效

Agent 持有连接的时间 = 查询执行时间 + LLM 推理时间 × 推理步数。单个任务可能分叉出多个子 Agent，并发 Agent 数量可能远超开发预期。

**应对策略**：
- 为 Agent 工作负载设置独立的连接池
- 使用 PgBouncer 的事务池模式，让 20 个实际 Postgres 连接可服务 500 个 Agent 连接

```ini
# pgbouncer.ini
pool_mode = transaction
max_client_conn = 500
default_pool_size = 20
reserve_pool_size = 5
reserve_pool_timeout = 1.0
```

### 4. 错误可被察觉假设的失效

人类系统中慢查询会立即暴露问题。Agent 则可能基于错误查询的结果继续执行，产生难以察觉的语义级错误。

**应对策略**：通过 SQL 查询注释为每个查询附加 Agent 上下文信息。

```python
from sqlalchemy import text, event
from sqlalchemy.engine import Engine

@event.listens_for(Engine, "before_cursor_execute")
def add_agent_context_comment(conn, cursor, statement, parameters, context, executemany):
    agent_ctx = getattr(conn.info, "agent_context", None)
    if agent_ctx:
        statement = f"/* agent_id={agent_ctx['agent_id']}, task_id={agent_ctx['task_id']}, step={agent_ctx['step']} */ {statement}"
    return statement, parameters
```

### 5. Schema 是开发者契约假设的失效

Agent 通过 Text-to-SQL 或 MCP 服务器访问数据库时，列名和表结构直接影响查询质量。遗留系统的命名（如 `stat_cd`、`flg_1`）会导致 Agent 生成"自信的错误"查询。

**应对策略**：为 Agent 构建专用视图层，并用类文档字符串的注释描述列含义。

```sql
COMMENT ON COLUMN agent_orders.fulfillment_status IS
  'Current state of the order in the fulfillment pipeline. '
  'Use this to filter orders that need action: pending and processing orders are active. '
  'Cancelled orders should never be modified.';
```

### 6. 爆炸半径控制

Agent  Reasoning 出错或凭证泄露时的最大损失由其被授予的访问权限决定。

**应对策略**：每个 Agent 类型使用独立数据库角色，执行最小权限原则。

```sql
CREATE ROLE agent_fulfillment;
CREATE ROLE agent_customer_support;
CREATE ROLE agent_analytics;

GRANT SELECT, UPDATE (fulfillment_status, shipped_at, tracking_number)
  ON orders TO agent_fulfillment;
```

## 总结

本文的核心观点：传统数据库架构的五大假设（确定性调用者、意图性写操作、短暂连接、明显故障、Schema 即开发者契约）因 AI Agent 的引入而全面失效。这些原本被视为"可选的最佳实践"的模式——软删除、仅追加日志、幂等性键、最小权限角色、查询标记——现在已成为"支撑性基础设施"。

防御性数据层的关键要素：
1. 每个 Agent 类型拥有独立数据库角色和超时设置
2. 独立于人类流量的 Agent 专用连接池 + PgBouncer 事务模式
3. Agent 可写的表使用软删除并记录删除者身份
4. 高风险写路径使用仅追加事件日志 + 幂等性键约束
5. Agent 可见的 Schema 采用可读命名 + 文档化注释
6. 每条查询携带 Agent ID、任务 ID、推理步骤的标记

**核心理念**：将数据库视为防御层，假设调用者可能是错的、可能重试、可能没有监控结果。
