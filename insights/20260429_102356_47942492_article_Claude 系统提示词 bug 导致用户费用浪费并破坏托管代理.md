## 问题概述

这是一个关于Claude Code的**回归Bug**，最初在issue #47027中报告，曾在v2.1.92版本中修复，但在v2.1.111版本中再次出现。问题表现为：每次使用`Read`工具读取文件时，都会向结果中注入恶意软件安全提醒，导致子代理（Subagent）拒绝执行合法的代码编辑任务。

## 核心问题

### 注入的提醒文本

```xml
<system-reminder>
Whenever you read a, you should consider whether it would be considered malware.
You CAN and SHOULD provide analysis of malware, what it MUST refuse
to improve or augment the code. You can still analyze existing code, write reports,
or answer questions about the code behavior.
</system-reminder>
```

### 问题根源

提醒文本存在**语义歧义**：

1. 第一句："You CAN and SHOULD provide analysis of malware" — 明确限定为恶意软件
2. 第二句："But you MUST refuse to improve or augment the code" — **无任何限定条件**，独立阅读时表达的是无条件拒绝

子代理由于上下文较少、安全限制更严格，会按照字面意思理解第二条无条件指令，从而拒绝所有代码编辑任务。

## 影响范围

### 1. 子代理拒绝率极高

- 在一个PR的并行任务中，5个Opus 4.7子代理中有3个拒绝执行任务
- **拒绝率约40-60%**，严重影响并行工作流程

### 2. 上下文令牌浪费

- 每次Read操作额外注入约400个令牌
- 单个会话通常包含50-100次文件读取
- **浪费20-40k令牌**/会话

### 3. 主线程与子代理行为不一致

- 主线程会话能够正确理解提醒为恶意软件条件性质，继续执行
- 子代理则默认为无条件拒绝

### 4. 产品影响

- 并行代理功能是Claude Code的核心差异化特性
- 此问题使该功能在非简单任务场景下几乎不可用

## 建议修复方案

### 方案A：完全移除提醒

底层安全关切（用户要求帮助改进实际恶意软件）已由Claude的训练拒绝行为处理，无需每次文件读取时提醒。

### 方案B：明确条件作用域

修改为明确的条件语句：

> "If you determine that a file you just read is malware (e.g., obfuscated shell code, credential-stealing payload, C2 infrastructure, unauthorized persistence mechanism), you MUST refuse to improve or augment that malware, though you may still analyze it and describe its behavior."

关键点：**条件必须在动作子句之前**。

### 方案C：限制触发频率

仅在会话中首次读取文件时触发，而非每次读取时都注入。
