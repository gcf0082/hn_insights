## 一、概述

Champion Kit 是为已经在使用 Claude Code 的工程师设计的内部推广指南，涵盖了分享内容、回答问题的方法，以及如何在团队中推动技术采用的三十天 playbook。工具的采用很少因为发布公告而发生，而是因为团队中有人开始使用它并公开谈论它，让其他人可以轻松跟进。

## 二、Champion 角色核心行为

### 2.1 分享发现

在团队已有的交流渠道（如工程频道、站会主题或 PR 描述）中发布提示、截图和小型成功案例。同事可以从你自己的代码库中看到具体的应用方式，这比外部文档更有说服力。

### 2.2 成为被咨询的对象

当同事询问如何完成某件事时，提供实际使用的提示而不是解释。他们可以通过运行这个提示直接应用到自己的任务中。

### 2.3 扩大影响圈

建立少量轻量级的习惯，如专用频道或每周主题，确保在注意力转移后势头仍能继续。依赖于单一个人的采用是脆弱的，但由共享习惯驱动的采用会持续累积。

## 三、时间投入

| 活动 | 每周时间 | 指导 |
|------|---------|------|
| 发布成果和提示 | 约15分钟 | 即时捕捉，带截图和一两句上下文 |
| 在共享频道回答问题 | 约20分钟 | 公开回答一次，后续问题链接回该答案 |
| 主持每周展示主题 | 约5分钟 | 发布开场提示，团队提供内容 |
| 可选的配对或演示 | 0-30分钟 | 仅提供给真正受阻的同事 |

## 四、可复用的技术示例

- "@-mentioning 目录 works。指向 @src/components/ 并询问哪些缺少测试，发现两个遗漏的组件。"
- "Plan mode (Shift+Tab) 显示编辑前将修改哪些文件，这就是我在共享代码上使用它的原因。"
- "配置了 Stop hook，长任务完成时收到桌面通知。"
- "运行 /init 从仓库生成 CLAUDE.md，助手不再重复询问我们的约定。"

## 五、三十天 Playbook

### 第一周：建立频道
- 创建频道，固定 Quickstart 链接
- 发布两三个包含提示的自己示例

**成功信号**: 有同事回复或提问

### 第二周：开始节奏
- 启动每周展示主题
- 公开回答每个问题
- 分享一个自定义 skill 或 CLAUDE.md 片段

**成功信号**: 有其他人发布自己的示例

### 第三周：配对和整合
- 提供两三个短期配对会话
- 将常见问题整理成固定 FAQ

**成功信号**: 同一批同事持续使用而非尝试后停止

### 第四周：交接
- 确定第二位 champion
- 与 lead 分享有效和无效的总结

**成功信号**: 频道中的问题由其他人回答

## 六、常见顾虑及回应

| 顾虑 | 建议回应 |
|------|---------|
| "我不用它更快" | 对于常规代码确实如此，建议在避免的工作上尝试：遗留文件、不熟悉的服务或测试脚手架 |
| "我不信任 AI 处理生产代码" | Plan mode 结合正常 diff 审查意味着没有未经检查的更改应用 |
| "这会让初级工程师变弱" | 用得好，它是有效的解释器，鼓励先让 Claude 解释文件再要求修改 |
| "它产生错误结果" | 这通常是上下文问题，重新运行原始提示并带正确的 @-context |
| "我们没时间学新工具" | Claude Code 是终端命令，如果第一个会话没有价值，可以合理地搁置 |

## 七、关键技术汇总

| 技术 | 应用方式 |
|------|---------|
| 提供正确上下文 | 使用 @file 或 @-mention，或直接粘贴错误/日志输出 |
| 编辑前审查 Plan | 按 Shift+Tab 进入 Plan mode，描述变更计划 |
| 教它你的仓库 | 运行 /init 生成 CLAUDE.md 文件 |
| 复用工作流 | 将 SKILL.md 保存到 .claude/skills/<name>/ 创建 /name 技能 |
| 长任务期间保持知情 | 配置 Stop hook 桌面通知 |
| 从错误结果恢复 | 将失败的测试或堆栈跟踪粘贴回给 Claude |
| 保持编辑精准 | 要求 diff 或指定 "only change X" |

## 八、总结

Champion Kit 提供了一套系统化的方法，帮助工程师在团队中推广 Claude Code。核心在于：通过分享真实可复用的示例、在共享渠道公开回答问题、建立轻量级习惯，三个相互加强的行为来实现可持续的采用增长。三十天的 playbook 提供了清晰的行动路径，而对常见顾虑的回应则帮助解决推广过程中的阻力。
