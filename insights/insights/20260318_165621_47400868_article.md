## 项目概述

**Godogen** 是一个基于 Claude Code skills 的 AI 游戏开发管道，能够从自然语言描述自动生成完整的 Godot 4 游戏项目。

核心特点：
- 端到端自动化：从描述 → 架构设计 → 资源生成 → 代码编写 → 视觉 QA
- 支持 2D 和 3D 游戏
- 可在普通硬件上运行

---

## 技术架构

### 双 Skill 编排系统

```
godogen (编排器)
    └── godot-task (任务执行器, context: fork)
```

1. **godogen** - 负责整体流程编排
   - 加载子模块文件处理各阶段任务
   - 管理任务状态和恢复逻辑
   - 通过 Telegram 向用户推送进度

2. **godot-task** - 负责单个任务执行
   - 在独立上下文中运行（避免状态污染）
   - 渐进式加载领域知识
   - 包含完整的 GDScript 参考和 850+ Godot 类 API

### 管道流程

```
用户请求 → 视觉目标 → 任务分解 → 架构设计 → 资源生成 → 任务执行 → 视觉QA → 演示视频
```

## 核心创新点

### 1. GDScript 深度适配

**痛点**: LLM 对 GDScript 训练数据薄弱，容易写出"看起来像 Python 但实际错误"的代码。

**解决方案**:
- 自定义 GDScript 语言参考 (`gdscript.md`)
- 完整 Godot API 文档（850+ 类）通过 sparse git clone 获取
- 两层懒加载索引系统：
  - `_common.md`: 128 个常用类
  - `_other.md`: 730+ 其他类
- Godot 引擎特性知识库 (`quirks.md`)

### 2. 场景构建器 vs 运行时脚本分离

两种代码类型的严格区分：
- **场景构建器**: 头less 模式运行，生成 `.tscn` 文件，不能使用运行时 API
- **运行时脚本**: 实际游戏逻辑，需正确处理 `set_script()` 和信号连接时机

### 3. 视觉质量保证循环

```
截图捕获 → Gemini Flash 视觉分析 → 与 reference.png 比对 → 自动修复(最多3轮)
```

QA 检查内容：
- Z-fighting、纹理拉伸、对象漂浮
- 渲染 bug：缺失纹理（品红色）、剔除错误
- 动态模式：卡顿、抖动、物理爆炸

### 4. 资源生成策略

- **Gemini**: 2D 艺术和纹理生成
- **Tripo3D**: 图片转 3D 模型（仅 3D 游戏需要）
- **预算感知**: 最大化视觉冲击力/成本比
- 精灵图生成、Alpha 通道移除、多层抠图

### 5. 文档协议

阶段间通过结构化文档通信：
- `reference.png`: 视觉锚点
- `PLAN.md`: 任务图谱
- `STRUCTURE.md`: 架构蓝图
- `ASSETS.md`: 资源清单
- `MEMORY.md`: 项目记忆

## 竞争优势分析

### 优势
1. **完整的开发管道**：不只是代码生成，而是从设计到测试的完整闭环
2. **视觉 QA**：自动捕获和验证游戏画面，捕获文本分析无法发现的 bug
3. **深度领域知识**：针对 GDScript 和 Godot 引擎的专门适配
4. **可恢复性**：文档化状态管理支持中途恢复
5. **成本优化**：预算感知型资源生成策略

### 局限性
1. 单次生成可能需要数小时
2. 需要 GPU 支持截图捕获
3. 动画生成仍是短板（依赖外部服务）
4. Godot 生态系统的依赖

### 与视频扩散方案对比
- **视频扩散**（Genie 等）：端到端学习，视觉逼真但逻辑不确定
- **Godogen**：确定性代码生成，逻辑正确但视觉需外部服务
- **混合可能**：Godogen 提供功能性游戏 + 视频扩散提供视觉增强

## 总结

Godogen 代表了 AI 辅助游戏开发的一种务实方法：通过深度适配 Godot/GDScript 生态、引入视觉 QA 闭环、以及模块化的 skill 编排系统，实现了从想法到可运行游戏的自动化。其核心创新在于将工程专业知识编码为可执行的提示规则，而非依赖纯学习的端到端方法。
