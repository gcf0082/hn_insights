# HN 洞察：Prompt Injecting Contributing.md

**来源**: [Hacker News](https://news.ycombinator.com/item?id=47441499)  
**日期**: 2026-03-20  
**原始链接**: https://glama.ai/blog/2026-03-19-open-source-has-a-bot-problem

## 主题概述

本文讨论了开源项目中 AI 机器人贡献者带来的问题。作者发现大量 AI 机器人向其项目提交 PR，但这些 PR 质量参差不齐，存在模板化、缺乏激情等问题。

## 核心发现

### 机器人识别结果
- **约70%** 的机器人在 PR 标题中按要求添加了 ROBOTS 标记
- **约30%** 的机器人没有标记自己 —— 这更令人担忧
  - 可能因为它们没有读取 CONTRIBUTING.md
  - 或者被指示忽略这些指令

### 机器人的 sophistication（ sophistication 程度）
- 部分机器人非常先进：跟进评论、响应审查反馈、遵循复杂指令
- 有机器人甚至完成了完整的 GitHub OAuth 流程、Docker 构建配置和构建验证

## 主要讨论观点

### 1. Prompt Injection 的本质
- 将任务委托给需要遵循外部指令的 agent 本质上就类似 `curl | bash`
- 区分在于：CONTRIBUTING.md 是授权的指令渠道，而工具模式描述是元数据

### 2. 防御的困难
- 无法依赖模型自我约束，因为攻击者控制提示词
- 真正的防御必须在权限/执行层面，而非推理层面
- 30% 不标记自己的数据表明：要么有明确指示忽略repo指南，要么根本没读取 CONTRIBUTING.md

### 3. 经济激励问题
- 为什么有人愿意花钱（token成本）发送这些无意义的 PR？
- 可能为了向潜在雇主展示"高 GitHub 活跃度"
- MCP 服务器作者自身使用自动化提交自己的服务器列表

### 4. 军备竞赛风险
- 一旦维护者开始用机器人过滤 PR，激励会反转
- 机器人作者会优化通过过滤而非写好代码
- 这与 SEO 垃圾 vs 搜索引擎的对抗类似，只是发生在代码库内部

### 5. 质量问题的讽刺
- 机器人不会运行本地测试、linting 或验证 CI 构建
- 这些对 agent 来说是trivial的任务
- 跳过这些是人为操作失败，而非技术限制

### 6. 社区影响
- 维护者容量 vs 贡献量严重不对称，且日益恶化
- 与机器人进行深入、思考性反馈后意识到对方永远不会跟进，这令人沮丧

## 有趣的观点

- **反向 CAPTCHA**：有人指出这相当于为 AI agent 发明的反向 CAPTCHA
- **代价信号理论**：蜜蜂的摇摆舞是代价高昂的信号，只有真正有价值的来源才会被传达——便宜信号（如简单"指向"）会被滥用
- **人与工具的界限**：我们传统上把计算机做的事当作自己做的（"我发了邮件"、"我浏览了网页"）

## 未解决的问题

1. 如何可靠地识别和过滤 AI 贡献？
2. 能否利用 prompt injection 让机器人做出真正有价值的贡献？
3. 当人与机器的界限模糊时，开源和我们的心理健康意味着什么？
