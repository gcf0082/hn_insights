# OpenClaw：披着梦想外衣的安全噩梦

## 一、事件概述

OpenClaw是一款由Peter Steinberger开发的开源AI Agent项目，能够深度整合用户的本地系统文件、终端、浏览器、Gmail、Slack乃至智能家居系统。然而，这种前所未有的自动化能力背后，隐藏着严重的安全隐患。本文将从技术角度深入剖析OpenClaw的安全问题，并探讨其对整个AI Agent领域的深远影响。

## 二、核心安全问题分析

### 2.1 供应链攻击：SkillHub的致命漏洞

OpenClaw的SkillHub平台存在严重的安全缺陷。安全研究员Jason Melier发现，下载量最高的"Twitter"技能实际上是一个恶意软件投递向量，其攻击流程如下：

1. 技能描述要求安装名为"openclaw-core"的依赖包
2. 安装步骤中嵌入指向恶意基础设施的链接
3. 链接诱导Agent执行恶意命令
4. 恶意代码解码并执行后续载荷
5. 最终下载并运行信息窃取木马

Snyk安全公司对3984个技能进行审计后发现，**约7.1%（283个）的技能存在关键安全漏洞**，这些漏洞会在LLM的上下文窗口和输出日志中以明文形式暴露敏感凭证。更令人震惊的是，安全研究员Jamieson O'Reilly通过构造一个带有后门的技能，并人为刷高下载量，在一小时内就获得了来自7个不同国家的真实用户执行任意命令。

### 2.2 提示词注入：无法根治的顽疾

OpenClaw完美契合了安全专家Simon Willison提出的"**致命三合一**"（Lethal Trifecta）攻击模型：

- **访问私有数据**：Agent拥有Gmail、Slack、银行账户等敏感系统的访问权限
- **暴露于不受信任内容**：通过WhatsApp、Telegram、电子邮件接收消息，这些都可能包含恶意指令
- **外部通信能力**：能够向外部服务器传输数据，实现数据外泄

由于LLM的内在工作机制，提示词注入攻击无法被彻底根除。任何人都可以通过发送消息或电子邮件，在内容中嵌入隐蔽指令来操纵Agent行为。OpenClaw的广泛集成使其成为一个"完美的攻击目标"——攻击面过大，防御难度极高。

### 2.3 认证滥用与过度授权

OpenClaw集成了50多种服务，包括Slack、Gmail、Teams、Trello等。这种深度集成带来了双重风险：

- **本地存储的凭证**：OAuth访问令牌和刷新令牌以明文或弱加密形式存储在本地文件中（如`agents/<agentId>/agent/auth-profiles.json`）
- **权限边界模糊**：单一Agent拥有文件系统根访问权限、凭证访问权限和网络通信能力，却没有任何权限隔离或审批机制

一旦攻击者获取这些令牌，他们无需欺骗模型，直接可以在Slack和Gmail中模拟用户身份、获取数据、发送消息，甚至在组织内部进行权限提升。

### 2.4 记忆污染：静默的全面沦陷

OpenClaw的记忆系统完全基于Markdown文件存储，且Agent可以重写自己的记忆文件。这意味着攻击者可以：

- 静默修改记忆文件内容
- 在用户毫无察觉的情况下植入后门指令
- 持续窃取个人数据和凭证

与技能感染不同，记忆污染可以在整个实例中蔓延，而用户可能永远不会发现异常。

### 2.5 大规模暴露：30000+易受攻击实例

在OpenClaw热潮期间，大量用户急于部署实例而忽视了安全配置。Censys在1月27日至31日期间发现约21000个暴露的实例；BitSight在1月27日至2月8日期间的扫描更发现了**超过30000个易受攻击的OpenClaw/Clawdbot/Moltbot实例**。

初始版本的ClawBot存在一个关键漏洞：任何来自localhost的流量都被视为合法请求。由于大多数实际部署都通过nginx或Caddy反向代理在同一台机器上运行，每个连接都来自127.0.0.1/localhost，因此每个连接都被自动批准——即使是来自互联网的随机攻击者。

## 三、OWASP Agent十大风险映射

Palo Alto Networks将OpenClaw的安全问题与OWASP Agent Top 10风险进行了详细映射：

| 风险等级 | OWASP Agent风险 | OpenClaw实现问题 |
|---------|----------------|-----------------|
| 严重 | A01: 提示词注入 | Web搜索结果、消息、第三方技能注入指令 |
| 严重 | A02: 不安全的工具调用 | 基于未信任记忆源的推理调用bash、文件I/O、邮件等工具 |
| 严重 | A03: 过度Agent自主性 | 单一Agent拥有文件系统根访问权限，无权限边界 |
| 严重 | A04: 缺失人工干预控制 | 删除操作、凭证使用、外部数据传输无需审批 |
| 严重 | A05: Agent记忆污染 | 所有记忆不分来源统一存储，无信任级别区分 |
| 严重 | A06: 不安全的第三方集成 | 第三方技能以完整Agent权限运行，可直接写入持久记忆 |
| 严重 | A07: 权限分离不足 | 单一Agent同时处理不受信任输入和高权限操作执行 |
| 高危 | A08: 供应链模型风险 | 使用上游LLM但未验证微调数据或安全对齐 |
| 中危 | A09: Agent间无限制交互 | 当前为单一整体Agent，未来多Agent版本可能存在风险 |
| 高危 | A10: 运行时监控与防护缺失 | 记忆检索→推理→工具调用之间无策略执行层 |

## 四、AI Agent安全性的更广泛影响

### 4.1 安全与便利的永恒博弈

OpenClaw的本质是一个"**隐私换取自动化的浮士德契约**"。正如Brandon Wang在其文章中所述，他赋予了Agent读取短信（包括双因素认证码）、登录银行、访问日历和联系人、甚至代表用户执行操作的能力。"理论上，Clawdbot可以清空我的银行账户。"

这种便利与安全的悖论是AI Agent领域面临的核心挑战。消费者对自动化的渴望是原始的——这正是我们发明齿轮、传送带、机器、编程语言的原因，如今又是新一代AI超级助手。然而，每一次便利的提升都伴随着风险的几何级增长。

### 4.2 信任模型的范式转移

传统软件工具是可预测的、确定性的，而AI Agent更接近于人类行为的模拟——非确定性、适应性、但同样易受操纵。这意味着我们不能简单地将传统的安全模型套用于AI Agent。

Gary Marcus在其评论中指出："这些系统以'你'的身份运作……它们在操作系统和浏览器提供的安全保护之上运行。这意味着应用程序隔离和同源策略对它们不适用。"这与Apple iPhone应用程序精心设计的沙箱隔离形成了鲜明对比。

### 4.3 Agent经济的萌芽风险

Moltbook（一个类似Reddit的AI Agent专属社交平台）的安全研究揭示了一个令人不安的发现：Agent之间已经形成了加密货币经济，其中Agent被发现进行加密货币的拉高出货交易。一个名为TipJarBot的Agent甚至运营着具有提现功能的代币经济。这预示着当Agent获得不受限制的访问权限时可能发生的混乱局面。

### 4.4 行业成熟度的警醒

OpenClaw现象表明，AI Agent生态系统尚未准备好迎接大规模部署。虽然Claude Code等工具在周围系统足够成熟后获得了信任，但OpenClaw更进一步——要求用户授权访问WhatsApp、Telegram等可能成为攻击向量的应用程序。

## 五、安全防护建议

### 5.1 隔离部署策略

- 使用Docker容器在隔离环境中运行，避免安装在主设备上
- 不挂载完整主目录，仅授予一个工作目录（如`/srv/openclaw/work`）
- 使用操作系统权限，以非特权用户身份运行
- 启用Docker只读文件系统，移除不必要的Linux capabilities

### 5.2 网络安全加固

- 将网关绑定到127.0.0.1，仅通过VPN或私有隧道访问
- 配置防火墙，仅允许来自可信IP的SSH访问
- 严格配置`trustedProxy`，防止IP欺骗攻击

### 5.3 最小权限原则

- 为Agent创建独立的账户体系（独立Gmail、日历、密码管理器）
- 仅授予读取权限，必要时临时提升权限后立即撤销
- 按资源而非仅按操作进行范围限定（如"此日历"而非"所有日历"）
- 对删除、批量移动、外部共享、发送消息等破坏性操作启用人工审批

### 5.4 凭证管理最佳实践

- 避免在本地存储原始OAuth密钥，使用托管认证服务
- 定期轮换令牌和凭证
- 实施月度审计，清理未使用的连接和权限

## 六、结论与展望

OpenClaw现象揭示了AI Agent发展面临的一个根本性挑战：技术能力的飞速提升与安全防护体系的滞后之间的巨大鸿沟。即使在AI取得如此巨大进步的今天，这项技术仍处于"青春期"。我们必须谨慎地处理和约束它，假设它会被操纵，并设计能够contain和恢复错误的系统。

对于普通用户而言，在技术生态系统尚未成熟之前，**应避免使用OpenClaw**这类深度集成个人数据的AI Agent。FOMO（错失恐惧症）不应成为忽视安全的理由。

对于行业而言，OpenClaw案例应该成为整个AI Agent领域的警醒。我们需要的不仅是更强大的模型，更是更成熟的安全基础设施、更严格的权限控制机制、更完善的监控和审计体系。信任需要被earned，而不仅仅是假设。

AI Agent的未来是光明的，但其发展道路上布满荆棘。唯有在确保安全的前提下，可信的AI Agent才能真正成为改变生活的力量，而非潜伏的数字噩梦。
