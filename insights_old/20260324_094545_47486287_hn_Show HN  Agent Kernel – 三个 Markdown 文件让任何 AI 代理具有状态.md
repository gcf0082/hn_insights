# Agent Kernel 评论分析

## 支持的观点

### 1. 文本文件 + Git 的方案值得认可
多位开发者对这种简单直接的方案表示肯定，认为纯文本文件配合Git进行版本控制比向量数据库更加轻量和实用。

### 2. 适用于创建专业化 Agent
有评论者指出，这种方法非常适合创建多个副本并分别用作专业化 Agent，无需使用 OpenClaw 来管理邮件、日历、Slack、待办事项等任务。Agent Kernel 具备人格、持久记忆和自我修改能力，使用 Skills 与使用 Claude Code 的 Skills 类似。

### 3. 项目文档结构合理
有人分享了类似的文档组织方式：项目文档应包含 project.md（记录架构、责任图、功能和部落知识）和 TODO.md（记录差距、任务和进度，按功能分组），不建议在其他地方写文档。

### 4. HTTP 代理方案值得关注
有开发者正在开发类似 Spotless 的工具，采用 HTTP 代理方式让状态管理成为 Agent 无需担心的事情，该工具直接读写往返 Anthropic 的消息数组。

## 反对的观点

### 1. 方法可靠性存疑
有开发者表示之前尝试过类似方案，发现这种方法并不真正可靠。

### 2. 现有工具已有类似功能
评论指出 Claude Code 等工具已经具备自动记忆功能，这类方案可能只是重复造轮子。

### 3. Agent 无法可靠遵循指令
多位开发者强调，随着上下文增大，Agent 不能总是可靠地遵循 AGENTS.md 文件中的指令。如果要保证某些事情发生，应该使用 hooks。

### 4. Agent 不擅长判断何为重要信息
这是反馈最集中的问题。Agent 无法理解什么值得"记住"，会创建大量冗余内容，需要手动清理。

### 5. 扩展性问题
随着 notes/ 目录在数周或数月内增长，读取过去的历史记录会占用大量上下文窗口。

### 6. 追加日志的副作用
append-only 的日志会记录失败尝试，Agent 可能重复过去的错误方式而非避免它们。即使只遗漏一条引用旧方式的注释，Agent 也会创建整套转换系统。

### 7. Agent 会产生偏见
Agent 读取大量抱怨性日记后会自动站在用户一边，成为"点头先生"。有开发者不得不专门在规则文件中加入"考虑对方观点"的提示。

### 8. 创伤重播现象
Agent 会记住并重复之前的失败尝试，而非避免重蹈覆辙，导致陷入循环。

### 9. 本质上只是 Prompt
有评论直指核心——这本质上就是 prompt，要求 Claude Code 自己写这些文件同样有用。

## 普遍达成一致的观点

### 1. 文本文件 + Git 是更好的方案
普遍认为纯文本文件配合 Git 比向量数据库更好用。

### 2. 手动管理很重要
重要的决策和学习应该由开发者手动记录，不应完全依赖 Agent 决定什么值得记住。Git 本身就能提供大部分所需信息。

### 3. AGENTS.md 应保持最小化
近期研究表明，过多的指令文件反而可能负面影响输出，应尽量精简。

### 4. 短期或中期记忆更实用
多位开发者表示只使用短期或中期会话日志或任务文档，不依赖长期记忆。

### 5. 子 Agent 信息传递是难点
关于子 Agent 如何记录工作、 orchestrator 如何处理子 Agent 返回结果方面，仍存在挑战。

## 总结

评论总体上对 Agent Kernel 的创新表示认可，但实战经验丰富开发者指出了多个实际运用中的问题，尤其是 Agent 对"记住什么"的判断能力不足、长期记忆带来的偏见风险，以及扩展性问题。多数人认为纯文本加 Git 的方案可行，但关键在于保持人工干预和最小化指令文件。
