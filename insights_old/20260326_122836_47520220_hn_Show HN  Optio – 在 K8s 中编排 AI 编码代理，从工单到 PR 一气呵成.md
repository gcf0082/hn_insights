## 项目概述

Optio 是一个开源编排系统，利用 AI 编码代理将 GitHub Issues 工单自动转化为已合并的 Pull Request。用户只需指向其代码仓库，系统即可处理整个生命周期：

- ** Intake（接入）**：从 GitHub Issues、Linear 或手动创建任务
- ** Execution（执行）**：为每个仓库启动隔离的 K8s Pod，在 git worktrees 中运行 Claude Code 或 Codex
- ** PR 监控**：每 30 秒检查 CI 检查、审查状态和合并就绪情况
- ** Self-healing（自愈）**：在 CI 失败、合并冲突或审查者变更请求时自动恢复代理
- ** Completion（完成）**：Squash 合并 PR 并关闭关联的 Issue

核心创新在于反馈循环机制。当 CI 失败时，Optio 会将错误信息反馈给代理；当审查者请求变更时，这些评论会成为代理的下一个提示词。系统会持续运行直到 PR 合并或用户手动停止。

技术栈：Fastify、Next.js、BullMQ、Drizzle on Postgres，提供 Helm Chart 用于生产部署。

## 社区讨论焦点

### 1. 代理重试机制
有用户询问如何处理代理的重试问题，因为重试会消耗大量 token。开发者回复称，只要代理能够"碰壁"并看到错误，它们通常能够自我纠正。

### 2. 多租户隔离
社区关注是否适合多租户场景。当前的实现没有特殊的沙箱隔离，代理可以正常访问工具和进行网络调用。开发者表示正在考虑添加禁用外部网络连接的选项。

### 3. 代码质量保障
针对"如何防止代理产生破坏代码库的垃圾代码"的问题，开发者指出三点保障：
- 配置 CI/build 检查在 GitHub 运行，代理必须确保通过才能合并
- 可配置审查代理，按需确保遵循特定规则
- 用户可以禁用自动合并设置，自己审查所有代码

有评论者提醒，对 LLMs 应避免使用绝对化的语言如"确保"，审查代理提供的只是语义层面的约束，而非硬性保证。

### 4. MCP 支持
社区询问是否支持 Skills 和 MCP（Model Context Protocol）。

### 5. 实际应用案例
开发者分享了使用该工具完成的真实项目：在 Rust 中重新实现 curl/libcurl，通过子代理并行处理各个失败测试，最终完成整个项目。

## 总结

Optio 代表了 AI 编程助手机器人流程自动化的新尝试，将任务分配、执行、监控和自愈整合到一个完整的闭环系统中。尽管社区对多租户安全隔离和代码质量保证仍有顾虑，但作者以实际项目（Rust curl 实现）证明了工具的可行性。该项目体现了 AI 代理从"单次对话"向"持续自动化流程"演进的趋势。
