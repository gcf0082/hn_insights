# 构建数字门童：基于AI Agent的个人作品集助手

## 概述

本文介绍了一个创新的AI Agent项目——"数字门童"。作者在每月7美元的VPS上部署了一个AI Agent，连接到自建的IRC服务器，并将其指向自己的GitHub仓库。访客可以向它询问关于作者作品的问题，获得由实际代码支撑的回答，而非经过重新表述的简历文本。

## 核心问题与解决方案

### 传统作品集网站的痛点

大多数带有AI聊天机器人的作品集网站采用相同模式：将简历输入模型，让访客重新表述。这是一种"戏法"，模型无法告诉你任何简历中未包含的内容。

### 创新解决方案

作者构建了一套基础设施：当 hiring manager 问"George如何处理测试覆盖率？"时，答案不应是"George重视全面测试"，而应克隆仓库、统计测试数量、读取CI配置，然后给出具体答案。

## 系统架构

系统包含两个Agent和两个独立的服务器：

### 公开层：Nullclaw

- 运行在最小化的边界服务器上
- 678 KB的Zig二进制文件，内存占用约1 MB
- 负责问候、根据预加载的上下文回答问题
- 可克隆仓库以真实代码佐证 claims

### 私有层：Ironclaw

- 运行在独立的、更高配置的系统上
- 可访问邮箱、日历和更深的个人上下文
- 处理从Nullclaw路由的复杂查询

### 通信路径

访客（浏览器）→ georgelarson.me/chat/ → Gamja Web IRC客户端 → Cloudflare（代理、TLS终止、机器人防护）→ Ergo IRC服务器 → #lobby频道 → Nullclaw（公开Agent）→ #backoffice频道（私有IRC频道，通过Tailscale）→ Ironclaw（私有Agent）

## 选择IRC的理由

1. **美学契合**：作品集网站采用终端UI风格，IRC客户端嵌入式设计符合品牌调性
2. **完全自托管**：Ergo IRC服务器、Gamja Web客户端、Nullclaw Agent都在自有基础设施上，无第三方API风险
3. **30年历史的协议**：IRC简单、成熟、无供应商锁定

## 模型选择策略

### 对话层：Haiku 4.5

用于问候、分诊、简单背景问题。亚秒级响应，成本极低。速度比深度更重要。

### 工具层：Sonnet 4.6（备用）

当Nullclaw需要克隆仓库、读取代码或综合多个文件的发现时使用。只在需要推理时才付费。

### 成本控制

- 每日2美元上限
- 每月30美元硬性上限
- 防止对话失控或滥用

### Portfolio信号

使用Opus做礼宾服务会传递错误信号。如果Haiku能处理，就不要发送给Sonnet。分层推理（便宜的处理路径，能干的处理重活）是保持每日2美元成本的关键。

## 安全态势

该服务器是公开的边界服务器，需要像边界服务器一样加固：

- **SSH**：非root用户、仅密钥认证、非标准端口、禁用root登录
- **防火墙**：UFW仅开放三个端口（SSH、IRC TLS、HTTPS）
- **Cloudflare代理**：Web访客从不直接接触服务器
- **Agent沙箱**：仅 supervised mode，仅限工作区文件访问受限命令白名单
- **成本控制**：每日2美元、每月30美元硬上限
- **审计日志**：每次工具调用都会被记录

哲学是最小化攻击面。服务器仅运行两个服务（ergo和nullclaw），不直接提供Web内容，且无法访问私有数据。

## 通信栈组件

- **Ergo**：IRC服务器，单个Go二进制文件，2.7 MB RAM
- **Gamja**：Web IRC客户端，152 KB构建大小
- **Nullclaw**：AI Agent运行时，4 MB Zig二进制文件，~1 MB峰值RSS

总占用：二进制文件小于10 MB，空闲时内存小于5 MB。

## 实际功能

- **"George使用什么语言？"**：不重复简历，可通过检查仓库验证
- **"他如何组织测试？"**：克隆仓库，读取测试文件，报告发现
- **"关于Fracture项目"**：从预加载的项目记忆中提取，可深入源码
- **"如何联系他？"**：提供联系方式，不幻觉电话号码
- **"可以预约通话吗？"**：Nullclaw通过Google的A2A协议调用Ironclaw处理请求

## A2A协议实现

Nullclaw已经支持Google的A2A协议（v0.3.0）：Agent卡片发现、JSON-RPC调度、任务状态机。实现了`a2a_call`工具，向远程Agent发送`message/send` JSON-RPC请求。

### 巧妙设计

Ironclaw上运行的Nullclaw实例没有自己的API key。相反，其LLM provider指向Ironclaw自己的网关作为透传：

```
Nullclaw (此服务器)
  └─ a2a_call工具 → POST /a2a
       └─ Ironclaw的Nullclaw（独立服务器，通过Tailscale）
            ├─ 接收A2A任务
            └─ provider配置："ironclaw" → http://127.0.0.1:3000/v1
                 └─ Ironclaw自己的网关
                      └─ 路由到Kilo → 实际LLM
```

一个API key，一个计费关系。拥有API key的Agent支付推理费用，无论谁发起请求。

## 安全防护

开放的A2A端点是提示注入面。因此Nullclaw有严格的防护：

- 仅特定请求类型可路由到Ironclaw：预约、可用性、联系方式
- 拒绝任意访客指令
- Ironclaw的A2A端点仅限Tailscale访问
- 两个Agent都以supervised mode运行，仅限工作区文件访问

## 关键收获

1. **模型选择与系统设计同等重要**：为每一层选择合适的模型是设计决策，不是设置开关
2. **Agent是最简单的部分**：通信栈、安全加固、DNS路由、TLS管理、Cloudflare集成比配置Agent花费更多时间
3. **IRC被低估**：1988年的协议是AI Agent的理想传输层
4. **Nullclaw和Ironclaw之间的分离是承重墙**：公开、最小化、可牺牲 vs 私有、有能力、受保护
5. **Agent间通信需要结构和可见性**：Google的A2A协议处理契约，Tailscale上的私有IRC频道处理审计追踪
6. **不要重复凭证**：透传模式意味着一个API key，一个计费关系，零凭证扩散
