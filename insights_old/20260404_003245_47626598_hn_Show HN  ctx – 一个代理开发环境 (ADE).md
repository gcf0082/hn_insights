## 项目概述

ctx 是一个代理开发环境（Agentic Development Environment，简称ADE），旨在为团队使用多个编码代理时提供更好的工作流管理。该工具由 luca-ctx 发布，提供了容器化工作空间、远程主机模型和本地合并队列等核心功能。

## 核心功能

### 1. 容器化工作空间
ctx 提供基于容器的隔离运行环境，解决了传统沙箱方案的局限性。开发者可以配置容器策略和网络策略，赋予代理更高的自主性，同时避免访问不安全的磁盘位置或互联网资源。

### 2. 多仓库支持
- **工作空间附件模式**: 当一个仓库为主、 others 为参考资料时，可使用工作空间附件
- **父目录模式**: 当工作涉及多个仓库时，可以直接在包含所有仓库的父目录初始化工作空间

### 3. 本地合并队列
ctx 的本地合并队列是其最重要的功能之一，工作流程如下：
1. 代理在独立的工作树中工作
2. 变更在隔离环境下验证通过
3. 提交到本地合并队列
4. 队列将变更重放至最新目标分支并运行验证
5. 若冲突或验证失败，合并被拒绝
6. 代理获取最新上游状态，解决冲突后重新提交

### 4. 跨代理协作
ctx 支持代理无关的多代理协作，例如 Claude Code 主代理可以调用 Codex 子代理。

## 社区讨论要点

### 是否开源
该项目最初在 GitHub 上显示为开源，但实际并非开源软件。作者表示这是借鉴 Claude Code 的做法，仅用于问题报告。

### 与现有工具的比较
与 Conductor 相比：
- ctx 提供 VM/容器隔离环境和明确的网络策略
- ctx 有本地合并队列用于多代理工作树之间的变更合并
- ctx 支持 Linux，并设计了"本地应用 + 远程 Linux 运行时"模式
- ctx 不仅仅专注于 Claude Code 和 Codex，支持更广泛的代理框架

### IDE vs ADE
作者认为 ADE 不应该是进行代码导航的主要场所。当需要深度代码导航时，最好的做法是在 IDE 中打开工作树。IDE 已经在导航和重构方面达到了世界级水平，没有理由在代理 UI 中重建这些功能。

### 沙箱与权限
关于"致命三要素"（Lethal Trifecta）的讨论：
1. 无限制的互联网访问是提示注入和数据泄露的途径
2. 代理可能访问或修改机器上不希望被触及的内容

ctx 通过容器和网络策略解决这些问题，与用户自行搭建 VM 的方案类似，但自动管理了许多 VM 复杂性，包括弹性内存。

### 本地模型支持
目前 ctx 没有专门为本地模型设置，但任何已经支持本地模型的代理框架在 ctx 中也能正常工作。Codex 和 OpenCode 都相对容易与本地模型配合使用。

## 用户反馈

### 积极反馈
- 工具设计深思熟虑
- 本地合并队列功能强大
- 多代理支持实用

### 问题反馈
- Linux 版本存在兼容性问题（启动后窗口空白，菜单栏灰色不可用）
- GitHub 仓库缺少源代码

## 总结

ctx 作为代理开发环境，专注于解决多代理协作工作流中的挑战，特别是合并队列和沙箱隔离方面。虽然不是开源软件，但其设计思路和功能实现为团队使用多个 AI 编码代理提供了有价值的参考方案。
