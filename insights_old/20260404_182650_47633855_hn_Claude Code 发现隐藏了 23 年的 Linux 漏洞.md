## 漏洞详情

据报道，Claude Opus 4.6 模型在 Linux 内核中发现了一个存在长达 23 年的安全漏洞。该漏洞位于 netfilter 模块中，具体问题如下：

- **漏洞类型**: 缓冲区溢出（buffer overflow）
- **根本原因**: 代码声明了一个 1024 字节的 owner ID 字段（这是合法但异常长的值），但实际使用的内存缓冲区只有 112 字节
- **触发条件**: 拒绝消息中包含 owner ID（最长可达 1024 字节），导致总消息大小达到 1056 字节，而内核将 1056 字节写入 112 字节的缓冲区

这与当年的 Heartbleed 漏洞如出一辙——某个人忘记了某个字段应该/可以有多长，而其他人也没有去检查。

## 社区讨论焦点

### 1. 关于"隐藏"的说法

部分开发者认为标题略有误导性：

> 不是"隐藏"，更可能是"没人愿意去检查"。

当设计协议或编写包含可变长度元素的代码时，"有效长度范围是多少？"应该始终摆在首位。

### 2. AI 代码审计的可行性

静态分析工具其实很容易发现这类问题，但问题在于：

- 没人运行这些工具
- 工具发现了但被淹没在大量误报中
- 发现了但被忽视

LLM 的优势在于其 100 万 token 的上下文窗口，使得"所有 bug 都是浅显的"。

### 3. 成本问题

关于使用 AI 进行代码安全的成本讨论：

- 通过 OpenRouter 使用 Sonnet：约 $0.001/消息
- 使用 Devstral 2512：约 $0.0001/消息
- 一次完整的代码审计（全面探索）：约 $5-750
- 深度分析复杂权限提升漏洞：约 $750

有人指出，推理成本在过去 3 年下降了 300 倍，没有理由认为这种趋势不会持续。中国模型（如 Kimi K2.5）也以更低的成本提供类似质量。

### 4. 质疑声

部分开发者对此持怀疑态度：

- 没有对比 GPT 5.4 或开源模型是否能发现相同 bug
- 这更像是 Claude Code 的广告
- 有人担心 AI 可能引入比发现更多的漏洞

### 5. 积极评价

尽管存在质疑，社区也给出了积极反馈：

- 代码审查是 LLM 的强项，尤其是 C++ 这种静态分析工具传统上产生大量误报的语言
- 即使一半时间是幻觉，找到那个真正棘手的 bug 也是值得的
- 人们对"氛围编程"（vibe coding）技术债务的兴趣超过了这个方向

---

## 总结

这次发现展示了 AI 在代码安全审计方面的潜力。虽然漏洞本身并不复杂（类似于 Heartbleed 的内存安全问题），但 AI 的全代码库上下文理解能力使其能够发现人类开发者多年未注意到的问题。成本正在快速下降，AI 代码审查可能成为未来的标准实践。
