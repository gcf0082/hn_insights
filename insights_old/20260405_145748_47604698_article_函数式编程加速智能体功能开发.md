## 核心观点

AI代理在生产环境中持续失败的根本原因并非模型能力不足，而是代码库架构问题。可变状态、隐藏依赖和纠缠的副作用使得代理输出具有非确定性且难以调试。**函数式编程是解决这一问题的答案**，这个答案早在40年前就已经存在。

---

## 问题分析

### 为什么AI代理会失败

1. **隐藏依赖**：代理读取一个"接收列表、返回列表"的函数，写测试，通过，但生产环境中失败，因为函数依赖于全局配置和数据库单例，而这些从未在函数签名中声明。

2. **代码库对自动化不友好**：
   - 全局配置对象存在于三个模块之外
   - 函数内部存在写磁盘的副作用
   - 测试 mock 的数据库与生产环境行为不同

3. **缺乏上下文**：人类开发者经过数月建立代码库的mental model，知道哪些函数会改变状态、哪些模块共享全局变量。而代理每次会话都从头开始。

### 行业现状

- Gartner预测：到2027年底，超过40%的代理AI项目将被取消
- MIT研究发现：95%的AI试点项目未能实现ROI

## 函数式编程解决方案

### 四个核心原则

1. **纯函数（Pure Functions）**：相同输入产生相同输出，无全局状态、数据库调用或日志

2. **显式数据流（Explicit Data Flow）**：输入如何变成输出可以通过线性阅读代码追踪

3. **边界处副作用（Side Effects at the Boundaries）**：I/O、数据库访问在薄薄的外层处理，核心逻辑是确定性的

4. **组合优于耦合（Composition over Coupling）**：小函数像乐高积木一样可以组合

---

## SUPER框架：五个代码原则

| 字母 | 原则 | 说明 |
|------|------|------|
| **S** | Side Effects at the Edge | I/O在边界层处理，不在业务逻辑内部 |
| **U** | Uncoupled Logic | 依赖通过参数传入，不从全局获取 |
| **P** | Pure & Total Functions | 确定性函数，处理每一种输入 |
| **E** | Explicit Data Flow | 数据线性流动，可追踪 |
| **R** | Replaceable by Value | 任何表达式可用其计算结果替换 |

### 实践示例

**修改前（有问题）：**
```python
def evaluate_options(options):
    threshold = GLOBAL_CONFIG["decision_threshold"]  # 隐藏依赖
    history = Database.get_instance().get_history()   # 隐藏依赖
    return [opt for opt in options if opt.score > threshold]
```

**修改后（SUPER合规）：**
```python
# 纯函数：所有输入显式，无副作用
def evaluate_options(options, threshold):
    return [opt for opt in options if opt.score > threshold]

# 边界：调用者处理I/O和配置
threshold = config.get("decision_threshold")
result = evaluate_options(orders, threshold)
```

## SPIRALS框架：七步过程循环

| 步骤 | 含义 | 说明 |
|------|------|------|
| **S** | Sense | 收集上下文：读取相关文件，检查git状态 |
| **P** | Plan | 制定方案，考虑权衡，定义"完成"标准 |
| **I** | Inquire | 识别知识缺口，验证假设 |
| **R** | Refine | 简化方案，应用80/20法则 |
| **A** | Act | 编写代码，遵循SUPER原则 |
| **L** | Learn | 运行测试，检查输出，记录失败原因 |
| **S** | Scan | 宏观检查：查找重复、新风险、可能的破坏 |

### 两个阶段

- **计划阶段**（S·P·I·R）：产生设计文档、票据和燃尽图，需要人类审核批准
- **执行阶段**（A·L·S）：每张票据独立运行：写代码→验证→检查回归→提交

## 实践效果

- 代理在SUPER合规代码上首次通过测试的频率是传统命令式代码的约3倍
- 调试时间大幅下降，因为失败只限于一个函数，而非跨共享状态图传播
- 使用SPIRALS后，代理需要的人工介入从持续监控减少到仅在Plan和Learn两步审核

## 实施建议

1. **找到修改最频繁的三个模块**：`git log --format=format: --name-only | sort | uniq -c | sort -rn | head -20`

2. **移除副作用**：将读取全局配置、访问数据库、写入日志等操作从函数内部移到调用者

3. **显式化依赖**：所有函数需要的值都应作为参数传入

4. **添加Scan步骤**：代理完成更改后，检查重复、潜在破坏、目标是否真正达成

5. **测量差异**：对比重构前后代理测试首次通过率
