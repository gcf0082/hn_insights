## 项目概述

Gemma Gem 是一个 Chrome 扩展程序，通过 WebGPU 在离屏文档中加载 Google 的 Gemma 4 (2B) 模型，并为其提供与任意网页交互的工具：读取内容、截图、点击元素、输入文本、滚动以及执行 JavaScript。

用户在每个页面都能获得一个小型聊天悬浮框。可以向它询问关于页面的问题，它（通常）能够自行判断应该调用哪些工具。它还具有"思考模式"，可以在工作时显示思维链（chain-of-thought）推理过程。

这是一个运行在浏览器中的 2B 模型，适用于简单的页面问题和 JavaScript 执行，但多步骤工具链的可靠性不足，有时它会完全忽略工具。Agent 循环没有任何外部依赖，如果有人想尝试，可以将其作为独立库提取出来。

## 社区观点

### 浏览器内置AI API的发展

有评论者指出，Chrome 还有一个正在处于 Origin Trial 阶段的 Prompt API，支持类似的 API 界面。一位用户展示了他使用 v3Nano 模型的统计数据：模型大小为 4,072.13 MiB，使用 GPU 作为后端。社区普遍认为，浏览器内置 AI 功能的原生化是未来趋势，但短期内还不会实现，因为模型下载量可能是浏览器本身的数倍。未来这些 API 可能会使用集成在操作系统中的 LLM，就像使用图形驱动程序一样。

Chrome 的 Summarizer API 已经发布，任何网站都可以通过简单的 `Summarizer.create()` 调用来静默触发 2 GB 的模型下载（需要用户激活）。

### 系统级AI推理的架构讨论

有评论者指出，从架构角度来看，在每个应用的用户空间中启动 LLM 是不合理的。既然现在有了专用的 NPU 和 GPU，需要一个统一的系统级编排器来平衡不同程序之间的推理队列——就像操作系统处理 NIC 或音频堆栈的方式一样。浏览器应该只是进行 IPC 调用，而不是拖着自己的重型推理引擎。

### 安全与稳定性考量

社区对安全性和稳定性表示担忧。有评论指出，让一个 2B 模型在实时页面上拥有完整的 JS 执行权限有点令人担忧。此外，为什么要将推理与浏览器生命周期绑在一起？如果 Chrome 崩溃或标签页被丢弃，代理的状态就消失了。一个带有"哑"扩展客户端的本地后台守护进程似乎更可预测和健壮。

但也有反驳观点：浏览器中有很多方式来存储数据，如 indexed db、OPFS 等，这些可以经受浏览器重启。后台守护进程除非用户自己安装和启动，否则无法工作，而这会带来很大的安装摩擦。浏览器应用的核心优势就在于无需安装东西。

也有评论者幽默地表示："每个我访问过的网页都有完整的 JS 执行权限，而我信任它们的程度比对 LLM 的信任还低。"

### 本地LLM的应用场景

有评论者表示希望看到有人将其构建为某种 SDK。应用开发者可以将其用作处理敏感数据时的本地 LLM 插件。通常让用户设置本地 LLM 的门槛太高，但这个问题可以得到解决。

也有用户展示了他在浏览器中运行小型嵌入模型的项目，虽然速度较慢，但可以做很多有趣的事情。
