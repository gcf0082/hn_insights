## 事件概述

2026年4月7日，Anthropic公司的Claude Code CLI工具出现宕机，用户无法通过OAuth登录，遭遇内部服务器错误。这是近期的又一次服务中断，引发了社区对AI编程工具稳定性的广泛讨论。

## 主要问题

1. **服务宕机**：Claude Code无法登录，OAuth认证失败
2. **状态页延迟**：status.claude.com 在用户报告后约11分钟才更新状态
3. **影响范围不一**：不同地区用户受影响程度不同（欧洲部分地区正常）
4. **桌面端可用**：Claude桌面应用正常，但Code CLI不可用

## 社区讨论焦点

### 1. 状态页透明度问题
社区强烈批评Anthropic的状态页未能及时反映服务状况。用户指出，状态页应该在真实指标基础上报告问题，而非"虚假宣传"。有人建议需要第三方独立监控服务来验证服务状态。

### 2. 宕机频率
有用户指出这几乎是每个周一早上都会发生的问题，怀疑是：
- 周一上班高峰期流量过大导致的扩展性问题
- 或故意限制免费用户以保证付费API用户的服务质量

### 3. 替代方案讨论
社区热议替代工具：
- **Qwen 27B**：有用户认为可与Sonnet媲美，且自托管更稳定
- **OpenAI Codex**：被推荐为优秀的替代方案
- **GLM-5**：作为备用方案
- **OpenCode**：也支持Claude模型

### 4. 供应商依赖担忧
用户表达了对单一AI编程工具依赖的担忧：
- "这次宕机让我意识到我对AI辅助编程的依赖程度"
- "开始对无法使用Claude感到焦虑"
- 有用户提到需要建立"模型无关"的开发工作流

### 5. 代码质量争议
有评论指出Anthropic虽然拥有强大的AI模型，但其自身系统存在大量bug，Claude Code仓库有5000+开放问题。这引发了对"AI公司是否真正使用自己产品"的讨论。

## 后续进展

- 约1小时后，服务逐渐恢复
- Anthropic最终更新了状态页，报告了服务问题
- 用户对官方最终公开透明地处理问题表示肯定

## 总结

此次事件反映了AI编程工具在企业级可靠性方面仍面临挑战。用户对状态页透明度、供应商锁定以及服务稳定性表达了强烈关切。社区开始探讨多元化工具策略，减少对单一供应商的依赖。
