## 项目概述

Kontext CLI 是一个用 Go 编写的凭证代理工具，专门为 AI 编码代理设计。当前大多数团队处理凭证的方式是将长期 API 密钥复制粘贴到 .env 文件或聊天界面中，这种方式存在以下问题：

1. **机密信息扩散**: 凭证分散各处，无法追踪
2. **缺乏访问 lineage**: 无法知道哪个开发者启动了哪个代理、访问了什么
3. **策略执行困难**: 一旦将原始凭证交给进程，就失去了强制执行策略、审计访问或轮换凭证的能力

## 核心功能

### 1. 凭证声明方式
在 `.env.kontext` 文件中声明项目需要的凭证：

```
GITHUB_TOKEN={{kontext:github}}
STRIPE_KEY={{kontext:stripe}}
LINEAR_TOKEN={{kontext:linear}}
```

### 2. 会话启动
运行 `kontext start --agent claude`，CLI 通过 OIDC 进行认证。

### 3. 凭证处理机制
- **对于支持 OAuth 的服务**: 通过 RFC 8693 token exchange 将会话替换为短期访问令牌
- **对于静态 API 密钥**: 后端直接注入到代理的运行时环境中
- **安全特性**: 凭证仅在会话期间存在于内存中，绝不写入磁盘

### 4. 审计能力
CLI 捕获每个工具调用的以下信息：
- 代理尝试执行的操作
- 执行结果
- 是否被允许
- 归属信息（用户、会话、组织）

### 5. 技术实现
- 使用 Go 编写（每次工具调用约 5ms hook 开销）
- 使用 ConnectRPC 与后端通信
- 认证信息存储在系统 keyring
- 当前支持 Claude Code，即将支持 Codex

## 与同类产品的比较

### 与 OneCLI 的区别
- **工作流**: OneCLI 作为自托管 Docker 网关运行，需要将代理流量路由到 localhost:10255；Kontext 不改变 Claude Code 的使用方式
- **可见性层**: OneCLI 拦截出站 HTTP 请求；Kontext 钩入 Claude 的 PreToolUse/PostToolUse 事件，可以看到 bash 命令、文件操作和 API 调用

### 与 Tailscale Aperture 的区别
Aperture 解决"让多个编码代理通过 Aperture 代理与正确的 LLM 后端通信"；Kontext 解决"启动具有身份、短期第三方凭证和工具级审计的受监管代理会话"。

## 社区反馈与讨论

### 主要关注点

1. **自托管问题**: 用户对凭证存储模式表示担忧，Kontext 在服务器端持有凭证并为每个会话生成短期令牌。许多人希望提供自托管选项。

2. **静态 API 密钥安全性**: 如果静态 API 密钥被注入到代理的环境中，代理理论上可以读取它。当前的威胁模型价值在于更好的保管、短期凭证（尽可能）和审计性，而不是"进程无法看到自己的环境"。

3. **服务支持限制**: 对于不支持 OIDC 的服务，这是一个已知的限制，用户只能获得较弱的安全层级。

4. **内存安全问题**: 有用户提问 Kontext 是否可以与 Claude 在同一用户下运行，理论上可以检查 Kontext 进程并从内存中提取密钥。
