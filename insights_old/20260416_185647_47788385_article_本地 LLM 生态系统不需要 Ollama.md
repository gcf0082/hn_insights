# Ollama 深度洞察报告：为什么不应再使用 Ollama

**洞察链接**: https://sleepingrobots.com/dreams/stop-using-ollama/  
**发布日期**: 2026年4月15日  
**作者**: Zetaphor  
**标签**: LLM, Local, AI, Open Source, Opinion

---

## 报告摘要

本报告揭示了 Ollama 项目的历史、技术问题及其商业模式转变。Ollama 作为最流行的本地大语言模型运行工具，本质上是一个基于 llama.cpp 的包装器，却多年拒绝承认其技术来源，业绩效低于开源引擎，且存在严重的闭源和云服务转向问题。

---

## 核心问题分析

### 1.  attribution 缺失与开源违规

Ollama 的全部推理能力来自 Georgi Gerganov 于2023年3月创建的 llama.cpp 项目。该引擎使得在消费级笔记本电脑上运行 LLaMA 模型成为可能，奠定了整个本地 LLM 运动的基础。

然而，Ollama 在其 README、网站或营销材料中**超过一年未提及 llama.cpp**。其二进制发行版也**未包含 llama.cpp 代码所需的 MIT 许可证版权声明**。

2024年初，社区提交了 GitHub issue #3185 要求许可证合规，**超过400天未获维护者回应**。直到社区 PR #3700 提出后，Ollama 联合创始人 Michael Chiang 才在 README 底部添加了一行说明。

社区回应揭示了真实态度：Ollama 团队表示正在“过渡到更系统性构建的引擎”，意图与 llama.cpp 保持距离。

### 2. 分支恶化与性能落后

2025年中期，Ollama 决定弃用 llama.cpp，转而基于 ggml（llama.cpp 使用的底层张量库）构建自定义后端。官方理由是稳定性需求。

结果却适得其反。Ollama 的自定义后端**重新引入了 llama.cpp 多年前已解决的 bug**，包括：

- 结构化输出支持损坏
- 视觉模型失败
- GGML 断言崩溃

在上游 llama.cpp 中正常运行的模型在 Ollama 中失败，包括 GPT-OSS 20B 等新版本。Georgi Gerganov 本人指出 Ollama 对 GGML 进行了错误的修改。

**性能对比数据**：

- 同一硬件同一模型：llama.cpp 运行速度比 Ollama 快 **1.8倍**（161 tokens/秒 vs 89 tokens/秒）
- CPU 模式下差距达 **30-50%**
- Qwen-3 Coder 32B 测试中，llama.cpp 吞吐量高出约 **70%**

性能开销来自 Ollama 的守护进程层、GPU 卸载策略不佳，以及落后于上游的后端。

### 3. 误导性模型命名

2025年1月 DeepSeek 发布 R1 系列模型后，Ollama 将较小的蒸馏版本（如 DeepSeek-R1-Distill-Qwen-32B，本质上是微调的 Qwen 和 Llama 模型）**简单标记为 "DeepSeek-R1"**。

用户运行 `ollama run deepseek-r1` 实际拉取的是 8B 的 Qwen 蒸馏模型，与真正的 6710亿参数 R1 模型完全不同。这并非疏忽——DeepSeek 本身已在命名中明确标注了 "R1-Distill" 前缀，Hugging Face 也正确列出了这些模型。

GitHub issues #8557 和 #8698 请求区分这些模型，均被关闭为重复问题。至今 `ollama run deepseek-r1` 仍启动一个微小的蒸馏模型。

### 4. 闭源应用发布

2025年7月，Ollama 为 macOS 和 Windows 发布了 GUI 桌面应用。该应用在私有仓库开发，**未发布许可证，源代码未公开**。

社区立即提出关切。许可证问题获得40票支持。开发者发现二进制中可能含有 AGPL-3.0 依赖。网站将下载按钮与 GitHub 链接并列，给用户造成下载的是 MIT 许可开源工具的印象。维护者沉默数月。

### 5. Modelfile：重新发明已解决的问题

GGUF 格式（Georgi Gerganov 创建）的核心设计原则是**单文件部署**——聊天模板、停止令牌、模型元数据全部嵌入文件内。

Ollama 添加的 Modelfile 是一种单独的配置文件，用于指定基础模型、聊天模板、系统提示、采样参数和停止令牌。其中大部分信息已存在于 GGUF 文件中。

问题在于：

- Ollama 仅自动检测其已知列表中的聊天模板
- 若 GGUF 元数据中存在有效的 Jinja 模板但不匹配 Ollama 已知模板，则静默回退到裸模板，破坏模型的指令格式
- 修改参数需要导出整个模型（30-60GB）才能更改一个参数

相比之下，llama.cpp 中参数是命令行标志：`--temp 0.7` 或 `--system-prompt "..."`。

### 6. 注册表瓶颈

新模型发布后，GGUF 通常在数小时内出现在 Hugging Face 上，由社区量化。使用 llama.cpp 可立即运行：`llama-server -hf unsloth/Qwen3.5-35B-A3B-GGUF:Q4_K_M`。

使用 Ollama 需要等待某人打包模型、选择量化方式、转换聊天模板并推送。通常仅提供 Q4_K_M 和 Q8_0，**不支持 Q5、Q6 或 IQ 量化**。

**量化限制**：Ollama 仅支持创建 Q4_K_S、Q4_K_M、Q8_0、F16 和 F32 量化。需要 Q5_K_M、Q6_K 或 IQ 量化（llama.cpp 已支持多年）的用户无法使用 Ollama。

### 7. 云服务转向

2025年末，Ollama 在本地库旁引入了云托管模型。这个曾被视为本地隐私推理代名词的工具**开始将提示路由到第三方云提供商**。

如 MiniMax 等专有模型出现在模型列表中，但选择这些模型会将数据发送到外部提供商。用户对数据路由提出担忧——Ollama 声称“我们处理您的提示和响应以提供服务，但不存储或记录内容”，但未说明第三方提供商如何处理数据。

2025年还发现了 CVE-2025-51471 漏洞——恶意注册表服务器可诱骗 Ollama 在正常模型拉取过程中向攻击者控制的端点发送身份验证令牌。

### 8. VC 模式分析

Ollama 是 Y Combinator（W21）支持的初创公司，由之前构建被 Docker Inc. 收购的 Docker GUI Kitematic 的工程师创立。

发展模式：

1. **基于开源 Launch**——基于 llama.cpp 构建，获得社区信任
2. **最小化 attribution**——使产品对投资者看起来自力更生
3. **创建锁定**——专有模型注册表格式、不可互操作的哈希文件名
4. **发布闭源组件**——GUI 应用
5. **添加云服务**——盈利模式

模型注册表使用 Ollama 自己的格式存储下载的模型，使用哈希文件名。用户无法直接将这些文件用于 llama.cpp 或 LM Studio。

## 替代方案推荐

### llama.cpp

真正的引擎，提供 OpenAI 兼容 API 服务器、内置 Web UI、完整的上下文窗口和采样参数控制，**性能始终优于 Ollama**。2026年2月，Gerganov 的 ggml.ai 加入 Hugging Face 确保项目长期可持续性。

### llama-swap + LiteLLM

处理多模型编排、加载、卸载和热切换，通过统一代理跨多个后端路由。

### LM Studio

提供 GUI，使用 llama.cpp 作为后端，支持任何 GGUF 模型，无锁定。

### Jan

另一个具有简洁聊天界面的开源桌面应用，采用本地优先设计。

### Msty

提供精美 GUI、多模型支持和内置 RAG。

### koboldcpp

提供 Web UI 和广泛配置选项。

### ramalama

Red Hat 的容器化模型运行器，明确列出上游依赖。**这正是 Ollama 从一开始就应做的事情**。

---

## 结论

Georgi Gerganov 在2023年3月的一个晚上hack出了 llama.cpp，掀起了本地 AI 的革命。数百名贡献者多年来致力于让消费者硬件上运行越来越强大的模型成为可能。这项工作真正重要，是保持本地推理开放和可访问的基础。

Ollama 将这项工作包装在一个不错的 CLI 中，凭借它获得 VC 资金，然后拒绝承认来源，糟糕地fork了它 alongside 推出了闭源应用，然后将整个产品线转向云服务。在每一个本可以成为优秀开源公民的决策点，他们都选择��让自己对投资者看起来更自力更生的道路。

**本地 LLM 生态系统不需要 Ollama。它需要 llama.cpp。其余都是包装，而更好的包装已经存在。**
