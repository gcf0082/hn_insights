## 核心观点

Ollama 是运行本地大语言模型最流行的工具，但作者认为它不应该这么流行。Ollama 通过成为第一个让 llama.cpp 变得易于使用的工具获得了关注，但该项目的成功建立在其多年系统性地隐瞒其技术真正来源、误导用户关于他们运行的内容，并逐渐背离使其获得信任的本地优先使命——同时还接受了风险投资。

## 详细分析

### 1. Ollama 的核心依赖问题

Ollama 的整个推理能力来自 **llama.cpp**，这是 Georgi Gerganov 于 2023 年 3 月创建的 C++ 推理引擎。Gerganov 的项目使得在消费级笔记本电脑上运行 LLaMA 模型成为可能，他在一个晚上就拼凑出了第一个版本。llama.cpp 如今在 GitHub 上拥有超过 100,000 颗星、450 多位贡献者，是几乎所有基于 GGUF 工具所依赖的基础。

然而，Ollama 的 README 在一年多的时间里**没有提及 llama.cpp**，既不在 README 中，也不在他们的网站或营销材料中。Ollama 的二进制分发也**没有包含 llama.cpp 代码所要求的 MIT 许可证通知**。

### 2. 许可证争议

社区注意到了这个问题。GitHub issue #3185 于 2024 年初开放，要求许可证合规。该 issue **超过 400 天没有收到维护者的回应**。当 issue #3697 于 2024 年 4 月开放 specifically 请求 llama.cpp 确认时，社区 PR #3700 在几小时内就提交了。Ollama 的联合创始人 Michael Chiang 最终只在 README 底部添加了一行：“llama.cpp project founded by Georgi Gerganov.”

### 3. 性能差距

基准测试显示了显著的差异。多个社区测试表明，llama.cpp 在相同硬件上运行相同模型时比 Ollama **快 1.8 倍**（161 tokens/秒 vs 89 tokens/秒）。在 CPU 上，差距是 **30-50%**。性能开销来自 Ollama 的守护进程层、糟糕的 GPU 卸载策略，以及落后于上游的封装后端。

### 4. 误导性的模型命名

当 DeepSeek 于 2025 年 1 月发布 R1 模型系列时，Ollama 将较小的蒸馏版本简单地列为"DeepSeek-R1"，而不是完整名称（DeepSeek-R1-Distill-Qwen-32B）。这导致用户在社交媒体上声称他们在消费级硬件上运行"DeepSeek-R1"，然后对性能不佳感到困惑，从而损害了 DeepSeek 的声誉。

### 5. 封闭式 GUI 应用

2025 年 7 月，Ollama 为 macOS 和 Windows 发布了一款 GUI 桌面应用。该应用在私有仓库中开发，没有发布许可证，源代码也不公开可用。这对于一个以开源为名声的项目来说是一个令人震惊的转变。

### 6. Modelfile 的问题

GGUF 格式的设计核心原则是单文件部署——模型文件中包含加载模型所需的所有信息。但 Ollama 在此基础上添加了 Modelfile，这是一个单独的配置文件。如果你想更改参数，工作流程是：使用 `ollama show --modelfile` 导出 Modelfile，编辑它，然后运行 `ollama create` 来构建新的模型条目。用户报告说这个过程会复制整个模型（30-60 GB），只是为了更改一个参数。

### 7. 注册表瓶颈

当新模型发布时，GGUF 通常在几小时内出现在 Hugging Face 上。使用 llama.cpp，你可以立即运行它们。但使用 Ollama，你需要等待有人为他们的注册表打包模型。这在 r/LocalLLaMA 上创造了 recurring pattern：新模型发布，人们通过 Ollama 尝试，结果很糟糕或很慢，而模型反而受到指责。

### 8. 云端转向

2025 年底，Ollama 引入了云端托管模型。这个与本地、私人推理同义词的工具**开始将 prompts 路由到第三方云提供商**。用户提出了关于数据路由的担忧——当通过"Ollama Cloud"运行 MiniMax-m2.7 等封闭式模型时，你的 prompts 可能会被转发到实际托管模型的外部提供商。

### 9. 安全漏洞

CVE-2025-51471 是一个 token 泄露漏洞，影响所有 Ollama 版本。恶意注册表服务器可以在正常的模型拉取过程中诱骗 Ollama 将其身份验证 token 发送给攻击者控制的端点。

## 推荐替代方案

作者认为，Ollama 所包装的工具可以直接访问，而且设置起来并不困难：

1. **llama.cpp** - 引擎本身，有 OpenAI 兼容的 API 服务器、内置 Web UI，持续改进性能
2. **LM Studio** - 如果你想要 GUI，它使用 llama.cpp，支持任何 GGUF 模型，无锁定
3. **Jan** - 另一个具有简洁聊天界面的开源桌面应用
4. **koboldcpp** - 具有 Web UI 和广泛配置选项的选择
5. **llama-swap** - 处理多模型编排、按需热插拔模型
6. **Msty** - 提供精致 GUI 和内置 RAG

## 结论

Ollama 包装了 Gerganov 的工作在友好的 CLI 中，基于它筹集 VC 资金，多年来拒绝给予信任，然后糟糕地分叉了它，在此过程中运送了封闭式应用，并将其转向云服务。在每一个本可以成为良好开源公民的决策点，他们都选择了让自己对投资者看起来更自立的道路。

**本地 LLM 生态系统不需要 Ollama。它需要 llama.cpp。剩下的就是包装，而更好的包装已经存在。**
