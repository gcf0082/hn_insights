## 核心概述

rtrvr.ai 推出的 AI Subroutines 技术是一种创新的浏览器自动化方案：用户只需在浏览器中录制一次任务操作，即可将其保存为可调用的工具。该方案的核心优势包括：

- **零 token 成本**：回放时无需调用 LLM，直接执行预生成的脚本
- **100% 确定性**：相同输入必然产生相同输出
- **认证自动传播**：在网页上下文中运行，cookies、CSRF tokens 等自动继承

## 核心问题：认证才是真正的难题

现有的 Web 代理方案只解决了问题的一半。要让 LLM 执行发推特、发私信、添加 LinkedIn 联系人等操作并不难，但当需要执行上千次时，成本问题就显现出来了：

- 每次调用消耗 token
- 每次调用存在延迟
- 结果并非总是一致的

看似简单的解决方案是直接调用网站的内部 API。然而这里有一个致命的难点：**认证**。

经过身份验证的网页请求需要携带多种认证凭据：

- Cookies
- 轮换的 CSRF tokens
- Session tokens
- Bearer headers
- Anti-replay nonces
- Fingerprint-bound 参数
- 由站点 JavaScript 计算的请求签名

这些认证信息有些是服务器设置的，有些是在浏览器中派生的，有些则是每次请求都会轮换。使用外部工具（如 Node workers、Playwright workers、cloud functions）需要在浏览器外重建所有认证机制，这正是大多数方案失败的原因。

## 核心架构：在扩展中录制，在网页内回放

rtrvr 的创新在于**录制和回放都发生在用户的浏览器内，从网页本身执行**：

1. **录制过程**：扩展拦截用户在执行任务时发出的网络请求，使用两层拦截机制：
   - MAIN-world 的 `fetch`/`XHR` 补丁：在任何页面脚本之前安装
   - Chrome 的 `webRequest` API 作为 CORS 和 service-worker 路径的备用方案

2. **回放过程**：脚本从页面的执行上下文分发——相同的域名、相同的 cookies、相同的 TLS 会话、相同的 JavaScript 计算签名_headers

这意味着：认证、CSRF、签名和指纹识别都自动传播，代理完全无需处理这些。

## 网络请求的排序和裁剪

除了"记录网络请求"之外，还有第二个问题隐藏其中。典型的一分钟浏览会触发数十到数百个请求——分析数据、RUM 请求、功能标志轮询、第三方像素、预取、媒体块等。用户关心的实际 API 调用往往淹没在噪声中。

因此，在生成器看到任何内容之前，rtrvr 会对捕获的请求进行评分和裁剪：

| 信号维度 | 权重 | 说明 |
|---------|------|------|
| 第一方 vs 第三方来源 | +20 / −15 | 已知的遥测主机（如 Sentry、Segment、Hotjar）直接 −80 |
| DOM 事件的临时关联 | +28（800ms内）/ +16（2.5s内）| 点击后80ms的 POST 与点击高度相关 |
| 方法和载荷形状 | +35（POST/PUT/PATCH/DELETE）/ +5（GET）| 带请求体 +8；OPTIONS/HEAD/perf entries −40 |
| 响应质量 | +12（2xx）/ −25（4xx+）| 非空响应体 +4 |
| 易失性操作标识符 | −18 | GraphQL queryId、doc_id、operationHash 等构建特定哈希 |

示例：第一方 mutating POST 在点击后80ms触发，响应200，得分约 +83。通用分析 beacon 得分 −80。

**关键机制**：如果最高分请求携带易失性操作标识符（如 X 的 queryId、Meta 的 doc_id），系统会强制生成纯 DOM 工具，并在生成指令中要求"不要暴露 queryId/doc_id/operationHash 值"。这是最有价值的故障案例：网络回放看起来很美好，但一周后网站部署新版本时就会悄然失败。

## Subroutines 作为工具调用

录制任务（称为 **Subroutine**）被注册为代理工具集中的可调用工具：

```
Instagram URL 列表    →  sendInstagramDirectMessage({ url, message })
每日内容队列         →  createXPost({ text, mediaUrl })
联系人列表          →  sendLinkedInConnectionRequest({ url, note })
```

优势：

- **零 token 成本**：回放是 `fetch`，不是推理
- **确定性**：相同输入，相同输出
- **低检测风险**：请求来自相同域名、相同请求头、相同的用户会话
- **LLM 可通过自然语言调用**

## 实现细节

- **参数绑定为命名函数参数**：Subroutine 体内运行在 `AsyncFunction` 中，参数为声明的参数加上 `rtrvr`。代码直接引用 `url` 和 `message`，不是通过字符串拼接注入。无模板注入漏洞。
- **`rtrvr.find` 遍历开放 shadow roots**：LinkedIn 的消息覆盖层和大多数 Material 3 组件在开放 shadow roots 中挂载交互元素， naive `querySelector` 会静默丢失。
- **`rtrvr.request` 自动处理 CSRF**：读取 cookie 中的 `ct0`/`XSRF-TOKEN`/`csrftoken`，注入 `x-csrf-token`，剥离回放敌对 headers（`set-cookie`、`x-client-transaction-id` 等）。

## 局限性

- **WebSockets/WebRTC**：执行 WebSockets、WebRTC 或繁重客户端派生的网站需要将 DOM 操作交织到 Subroutine 中，这是较慢且较不可靠的备选方案。
- **需要重新录制**：当网站显著更改 API 时需要重新录制。但实践中网站更改 API 的频率远低于 DOM。
