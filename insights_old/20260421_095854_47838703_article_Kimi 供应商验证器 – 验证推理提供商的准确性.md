## 为什么构建 KVV？

### 从孤立事件到系统性问题

自 K2 Thinking 发布以来，团队频繁收到社区反馈，称基准测试分数存在异常。经过调查确认，这些案例中有很大一部分源于**解码参数（Decoding Parameters）的误用**。

为解决这个问题，团队在 API 层面建立了第一道防线：
- 强制 Thinking 模式下 Temperature=1.0 和 TopP=0.95
- 强制验证 thinking content 正确回传

然而，更微妙的问题很快显现。在 LiveBenchmark 的特定评估中，第三方 API 与官方 API 存在显著差异。测试多个基础设施提供商后，发现这种差异是普遍存在的。

这暴露了开源模型生态系统的更深层问题：**权重越开放，部署渠道越多样，质量就越不可控**。如果用户无法区分“模型能力缺陷”和“工程实现偏差”，对开源生态系统的信任必然会崩溃。

---

## 解决方案：六项关键基准测试

KVV 选择以下六项基准测试来暴露特定的基础设施失败：

| 序号 | 基准测试 | 用途 |
|------|----------|------|
| 1 | Pre-Verification（预检验） | 验证 API 参数约束（temperature、top_p 等）是否正确执行。所有测试必须通过才能进行基准评估 |
| 2 | OCRBench | 5 分钟快速测试多模态管道 |
| 3 | MMMU Pro | 验证视觉输入预处理，通过多样化视觉输入测试 |
| 4 | AIME2025 | 长输出压力测试，捕获 KV cache  bugs 和量化退化 |
| 5 | K2VV ToolCall | 测量触发一致性（F1）和 JSON Schema 准确性 |
| 6 | SWE-Bench | 完整 Agent 编码测试（未开源） |

### 其他措施

- **上游修复**：与 vLLM/SGLang/KTransformers 社区合作，修复根本原因
- **发布前验证**：在用户遇到问题前，提供早期测试访问，让基础设施提供商验证其栈
- **持续基准测试**：维护公开的供应商排行榜，透明度鼓励供应商优先考虑准确性
