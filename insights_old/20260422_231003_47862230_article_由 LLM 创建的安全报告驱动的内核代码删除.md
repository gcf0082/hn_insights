# 洞察报告：LLM 生成的安全报告推动内核代码移除

**来源**: [LWN.net Article 1068928](https://lwn.net/Articles/1068928/)  
**发布日期**: 2026年4月22日  
**作者**: corbet  
**主题**: 内核代码移除（主要来自网络子系统）以应对 LLM 生成的安全报告激增

---

## 核心内容

Linux 内核维护者正在推进多项代码移除工作，主要涉及网络子系统中长期缺乏维护的驱动程序和协议栈。这些移除是为了应对大型语言模型（LLM）生成的安全漏洞报告数量急剧增加的问题。

### 提议移除的代码

1. **ISA 和 PCMCIA 以太网驱动程序**
2. **一对 PCI 驱动程序**
3. **AX.25 和业余无线电子系统**（包括 NET/ROM、ROSE 协议）
4. **ATM 协议和驱动程序**
5. **ISDN 子系统**

### 移除原因

维护者在补丁邮件中明确表示：

> "移除内核树中的业余无线电（AX.25、NET/ROM、ROSE）协议实现及所有相关无线电设备驱动程序。这组协议长期以来一直是巨大的 bug/syzbot 吸引器，由于没有人帮助我们处理AI生成的漏洞报告涌入，我们需要将其移出内核树来保护我们的 sanity。"

## 社区讨论焦点

### 1. 标题歧义与实际问题

社区评论指出，标题容易造成歧义——看起来像是 LLM 生成的安全报告帮助"移除"了内核代码，实际上是因为 LLM 报告的大量涌入才导致了代码移除。

讨论还指出，真正的问题在于大型项目（如内核）中大量未被维护的代码，它们通过成为大型项目的一部分来假装已被维护，而不是作为单独的项目让未被维护的状态早日可见。

### 2. 遗留硬件与维护责任

- **支持者认为**：这些是旧硬件设备，只需要随内核基础设施演进的变更，而非新功能
- **反对意见**：用户被"困在"旧内核上并不是好事，移除这些驱动会让用户无法使用新内核的安全更新
- **工业设备考量**：工业设备的生命周期以 decades 计，但制造商通常已倒闭，这些设备的价格通常是六位数（>= $100,000）

### 3. 安全的边界问题

针对"安全审计这些驱动程序的意義何在？"的讨论指出：
- 处理来自无线电天线数据的代码实际上应该比大多数其他代码更加硬化，因为任何人都可以发送数据
- 更大的担忧是，如果可以通过协议模块自动加载来加载模块，即使系统没有安装业余无线电设备，也可以利用其中的漏洞获取不应有的特权访问

### 4. Rust 重写建议

有建议认为可以用 Rust 重写这些驱动程序：
- **支持观点**：即使代码保持未维护状态，如果用安全 Rust 实现，也不会导致内核中的内存损坏
- **反对观点**： rewrite-and-forget 不是选项；内核新手/大学生需要愿意承诺维护这些驱动
- **实际考量**：并非所有安全 bug 都是内存损坏 bug

### 5. 用户空间迁移

社区普遍认为，许多业余无线电协议功能可以在用户空间实现：
- **Dire Wolf**：提供软件 TNC 功能，使用无线电的声卡接口
- **WSJT-X、JS8CALL、FreeDV**：已在用户空间运行良好
- **AGWPE**：AX.25 应用程序的当前技术状态

### 6. AI 工具的公平性

关于"如果 LLM 生成 bug 报告，也必须生成修复补丁"的讨论：
- 漏洞的存在与是否提供补丁无关
- 期望 AI 工具生成补丁而对其他工具（如 syzbot 或模糊测试器）有不同的标准是双重标准
- bug 和漏洞无论你是否喜欢所使用的工具都会存在

---

## 洞察总结

1. **根本原因**：LLM 显著提高了安全漏洞发现能力，导致遗留代码中的大量漏洞被发现，而缺乏维护者愿意修复

2. **权衡取舍**：移除未被维护的代码 vs. 继续维护产生的安全技术债务

3. **技术趋势**：部分功能正从内核空间迁移到用户空间，这是更合理的架构选择

4. **社区立场**：对于工业用途的遗留硬件，制造商应承担维护成本，而非志愿者
