# Claude Code 质量问题的深度分析报告

## 洞察基本信息

- **洞察链接**: https://www.anthropic.com/engineering/april-23-postmortem
- **发布时间**: 2025年4月23日
- **发布机构**: Anthropic
- **文章类型**: 事后分析报告（Postmortem）
- **语言**: 英文

## 报告摘要

Anthropic 发布了关于近期 Claude Code 质量问题的详细事后分析报告。报告追溯了过去一个月用户反馈的三个独立变更，这些变更导致 Claude 的响应质量下降。API 和推理层未受影响，所有三个问题已于 2025年4月20日（v2.1.116）修复完成。

## 事件背景

Anthropic 高度重视用户关于质量下降的报告。虽然 API 和推理层未受影响，但用户反馈的问题确实反映了实际存在的质量问题。经过深入调查，发现了三个不同的问题，每个问题影响不同的用户群体和时间节点，综合效果表现为广泛的、不一致的质量下降。

## 问题一：默认推理努力设置变更

### 事件时间线
- **2025年2月**: 发布 Opus 4.6 时将默认推理努力设置为 `high`
- **2025年3月4日**: 将默认推理努力从 `high` 改为 `medium`
- **2025年4月7日**: 恢复默认设置为 `high`

### 原因说明
部分用户反馈在高努力模式下，Claude 有时会思考时间过长，导致界面似乎冻结，产生不必要的延迟和 token 消耗。团队在内部测试中发现，中等努力模式虽然智能水平略低，但延迟显著降低，且不会遇到长时间延迟的尾部问题，有助于最大化用户的使用限额。

### 影响范围
- Sonnet 4.6
- Opus 4.6

### 后续措施
根据更多用户反馈，团队于 4月7日撤销了这一决定。现在 Opus 4.7 默认使用 `xhigh` 努力，其他模型默认使用 `high` 努力。用户可以自行调整推理努力级别。

## 问题二：缓存优化导致的推理丢失

### 事件时间线
- **2025年3月26日**: 部署缓存优化功能
- **2025年4月10日**: 修复 bug（v2.1.101）

### 技术细节
团队意图在会话空闲超过一小时后清除旧的思考内容，以减少用户恢复会话时的延迟。使用 `clear_thinking_20251015` API 头部和 `keep:1` 参数，只保留最近的推理块。

### Bug 机制
实现中存在一个关键 bug：原本应该只在会话恢复时执行一次的清除操作，实际在会话剩余的每个轮次中都执行了清除。这导致 Claude 越来越失去对之前决策的记忆，表现为健忘、重复和奇怪的工具选择。

### 影响范围
- Sonnet 4.6
- Opus 4.6

### 发现困难原因
两个不相关的实验使问题难以复现：
1. 一个内部-only 的服务端消息队列实验
2. 另一个变更掩盖了 CLI 会话中的 bug

Bug 跨越了 Claude Code 上下文管理、Anthropic API 和扩展思考的交叉点。即使经过多轮人工和自动化代码审查、单元测试、端到端测试和自动化验证，这个 bug 仍然通过了检查。加上问题只在特定角落案例（空闲会话）中发生，发现和确认根本原因花费了超过一周时间。

### 意外发现
在调查过程中，使用 Opus 4.7 的 Code Review 功能回测问题 pull request。当提供必要的代码仓库以获取完整上下文时，Opus 4.7 成功发现了这个 bug，而 Opus 4.6 未能发现。

## 问题三：系统提示变更导致的质量下降

### 事件时间线
- **2025年4月16日**: 随 Opus 4.7 部署新的系统提示
- **2025年4月20日**: 恢复原有提示

### 技术细节
Opus 4.7 相对于前代有一个明显的行为特点：倾向于更冗长。这使它在困难问题上更聪明，但也产生更多输出 token。团队尝试使用多种工具来减少冗长，包括模型训练、提示优化和改���思考 UX。

添加的系统提示指令：
> "Length limits: keep text between tool calls to ≤25 words. Keep final responses to ≤100 words unless the task requires more detail."

### 影响评估
经过数周的内部测试和评估，团队认为更改是安全的并随 Opus 4.7 一起发布。在后续调查中，使用更广泛的评估集进行消融研究（移除系统提示的各行以理解每行的影响），其中一个评估显示 Opus 4.6 和 4.7 的性能都下降了 3%。

### 影响范围
- Sonnet 4.6
- Opus 4.6
- Opus 4.7

## 改进措施

为防止类似问题再次发生，Anthropic 宣布将采取以下措施：

### 1. 内部测试流程改进
- 确保更大比例的内部员工使用与发布版本完全一致的 Claude Code
- 改进内部使用的 Code Review 工具并向客户发布

### 2. 系统提示变更管控
- 对每个系统提示变更运行针对各模型的广泛评估
- 继续进行消融研究以理解每行的影响
- 建造新工具使提示变更更容易审查和审计

### 3. 模型特定变更隔离
- 在 CLAUDE.md 中添加指导，确保模型特定的变更只影响其目标模型
- 对于可能影响智能的变更：添加浸泡期、更广泛的评估套件和渐进式推出

### 4. 用户沟通改进
- 在 X/Twitter 上创建 @ClaudeDevs 账号解释产品决策和背后的推理
- 在 GitHub 上集中分享更新

### 5. 用户补偿
- 为所有订阅用户重置使用限额

## 关键教训

### 发现模式
由于每个变更影响不同的时间段和不同的流量部分，聚合效果看起来像是广泛的、不一致的质量下降。这使得问题在早期难以与正常用户反馈变化区分，内部使用和评估也未能最初复现问题。

### 测试局限性
- 内部测试虽然通过但无法完全反映真实用户场景
- 特定角落案例（空闲会话）难以在测试中复现
- 多个独立变更的组合效果难以预测

### 透明度
Anthropic 选择公开这一事后分析报告，体现了对用户负责的态度。公开承认问题并解释根本原因是建立用户信任的重要步骤。

## 结语

这份事后分析报告展示了复杂软件系统中多个独立变更如何可能组合产生意想不到的负面效果。Anthropic 的公开回应和持续改进行动表明其对产品质量和用户信任的重视。三个问题的根本原因各不相同，但都指向同一个教训：在快速迭代的环境中，需要更严格的测试流程和更谨慎的发布策略。
