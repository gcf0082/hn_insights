## 项目概述

该项目名为 **WUPHF**，是一个面向 AI 代理的协作知识库，使用 Markdown + Git 作为数据持久化层，辅以 Bleve（BM25）全文检索和 SQLite 索引。项目理念源自 Karpathy 提出的"LLM 原生知识基质"概念——AI 代理既能从中读取也能向其写入，使上下文信息在多个会话间累积复用。

## 核心功能

1. **代理笔记本与团队 Wiki 分层架构**：每个代理拥有独立的笔记本（`agents/{slug}/notebook/.md`），内容经审核后可晋升为团队共享知识库（`team/`）
2. **实体事实日志**：以 append-only JSONL 格式记录实体事实，由"Pam the Archivist"这一固定 Git 身份提交，确保溯源可查
3. **Wiki 链接语法**：支持 `[[Wikilinks]]` 语法，断链以红色标记
4. **每日 lint 任务**：自动检测矛盾内容、陈旧条目和失效链接
5. **检索路由策略**：短查询走 BM25，长叙事查询走引用回答循环

## 技术选型亮点

- **BM25 优先于向量数据库**：在 500 条目、50 查询的基准测试中达到 85% recall@20，开发者认为大多数团队在未做性能评估前就贸然引入向量数据库
- **纯文本优先**：Markdown 文件完全开放、无厂商锁定，用户可随时迁出带走
- **SQLite 结构化元数据**：存储事实、实体、关系、重定向等，无需图数据库

## 社区讨论焦点

1. **笔记自动化的价值**：部分用户质疑自动化记录的效果，认为记笔记的核心价值在于阅读时的批判性思考，而非信息录入本身
2. **LLM 代理的可靠性**：有评论指出，代理运行时间越长失败概率越高，建议限制单次运行时间、分多次短时执行
3. **Karpathy 营销质疑**：有人吐槽"标题带上 Karpathy 就能获得关注"，也有人反驳这是合理的技术参考
4. **与 Obsidian 的对比**：被问及为何不直接用 Obsidian 插件，作者的回应是轻量级 Markdown + Git 更具可移植性
5. **BM25 与目录索引的选择**：有人质疑 BM25 的必要性，建议用自动维护的目录结构替代以提升精确率

## 总结

WUPHF 代表了一种"做减法"的 AI 知识管理思路：用最基础的 Markdown + Git 构建知识基底，BM25 满足检索需求，SQLite 处理结构化元数据，避免过早引入 pgvector、Neo4j、Kafka 等重型组件。对于追求简单、可移植、自托管的 AI 办公场景，这一方案值得关注。
