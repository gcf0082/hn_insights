## 项目概述

大多数 RAG（检索增强生成）系统的失败原因在于将记忆视为静态的文件柜。当每一个临时 bug 修复或被丢弃的规则都被永久存储时，上下文窗口最终会被噪声淹没，导致 token 成本飙升和代理推理能力下降。

该项目采用生物学方法，利用**埃宾浩斯遗忘曲线**将上下文管理为动态的 substrate。记忆被分配「强度」分数，每次检索都会强化数据并 flattening 其衰减曲线（间隔重复），而未使用的数据最终会达到阈值并被 pruning。

为了解决语义搜索遗漏相关但非相似节点的「逻辑邻居」问题，项目在向量存储之上叠加了图层。在 LoCoMo 数据集上的基准测试显示达到了 **52% Recall@5**，几乎是 stateless 向量存储准确率的两倍，同时减少了约 **84%** 的 token 浪费。

项目作为本地优先的 MCP 服务器使用 DuckDB 构建。其核心理念是：对于处理长期运行项目的代理来说，**「遗忘什么」与「记住什么」同样重要**。

---

## 核心创新

1. **生物_decay 机制**：引入埃宾浩斯遗忘曲线，模拟人类记忆的自然衰减
2. **强度评分系统**：每次检索强化记忆，flattening 衰减曲线（间隔重复原理）
3. **自动 pruning**：未使用数据达到阈值后自动清理
4. **图层面**：解决语义相似但实际不相关的「逻辑邻居」问题

## 社区讨论要点

### 反对观点

**SwellJoe** 认为记忆会使代理变笨：
> 我已经禁用了所有工具中的记忆功能。长期记忆让它们更笨更容易混淆。我不需要一个笨代理。
> 
> 更好的做法是让代理编写好的文档和开发计划，就像人类需要知道的那样。没有 MCP，没有技能，没有 API，只需要读取文件。这适用于任何代理、任何模型。

**pil0u** 指出记忆系统的 overhead：
> 我测试了几个「记忆」系统，但不确定它们是否提高了任务上的性能、质量或可靠性。但它们确实增加了开销。

**altmanaltman** 质疑「生物记忆」的概念：
> 整个「生物记忆」 thing 看起来像是基础缓存机制的营销术语。

**K0balt** 构建了类似系统：

## 关键洞察

1. **遗忘是 feature 而非 bug**：借鉴生物记忆的衰减机制，可能比永久存储更接近智能本质

2. **过度记忆的危害**：记忆系统可能让代理分心，基于之前的对话进行无效推断

3. **文档胜过记忆**：好的开发者文档和开发计划可能是更好的「记忆」形式

4. **性能 vs 复杂度**：52% Recall@5 和 84% token 节省的代价是系统复杂度的增加
